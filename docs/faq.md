# FAQ：常见问题解答

## 基础概念相关

### Q1: SD-WAN 和 VPN 有什么区别？

**A：** 这是初学者最常见的误解。

```
VPN（虚拟专网）：
  - 用途：加密通信
  - 工作层：应用层或网络层
  - 管理：通常由客户端软件完成
  - 例子：员工家里用 VPN 连接公司网络
  
  VPN 是"安全隧道"

SD-WAN：
  - 用途：广域网优化和管理
  - 工作层：网络层和传输层
  - 管理：由中央 Controller 完成
  - 例子：多个分支与总部的网络互连
  
  SD-WAN 是"智能网络"

关系：
  SD-WAN 内部可以使用 VPN（IPSec）来加密隧道
  VPN 可以运行在 SD-WAN 之上
  不是竞争关系，而是互补关系
```

### Q2: SD-WAN 是否一定要用云控制器？能本地部署吗？

**A：** 都可以，取决于企业的需求。

```
云端控制器（推荐）：
  优点：
  ✓ 高可用（多地冗余部署）
  ✓ 无需维护（厂商负责）
  ✓ 弹性扩展（自动扩容）
  ✓ 跨域管理（全球多地统一管理）
  ✓ 安全更新（自动获得最新安全补丁）
  
  缺点：
  ✗ 需要互联网连接到云平台
  ✗ 数据隐私（数据可能在云上）
  ✗ 成本（按年或月付费）
  
  适合：大企业、需要全球管理、接受云的企业

本地部署控制器：
  优点：
  ✓ 完全可控（数据完全在本地）
  ✓ 隐私性好（敏感数据不出本地）
  ✓ 无云依赖（即使互联网故障也能工作）
  
  缺点：
  ✗ 需要维护（需要专业运维人员）
  ✗ 需要高可用部署（主从冗余）
  ✗ 自动化程度低
  ✗ 成本高（硬件 + 人员）
  
  适合：金融、政府、需要绝对控制的企业

混合方案：
  主 Controller 在云端（日常管理）
  从 Controller 在本地（本地备份，可离线运行）
  → 既享受云的便利，又有本地控制
```

### Q3: CPE 一定要专用硬件吗？

**A：** 不一定，有多种部署选择。

```
硬件 CPE：
  - 独立设备，插在网络入口处
  - 优点：专业、可靠、性能高
  - 缺点：成本高、部署需要更改网络
  - 适合：大分支（>100 人）
  
  典型厂商：Cisco Meraki、Fortinet、Velocloud

软件 CPE（虚拟 CPE）：
  - 在 Linux 虚拟机或容器中运行
  - 可以部署在企业现有的路由器/防火墙上
  - 优点：成本低、灵活部署
  - 缺点：需要现有硬件支持、性能取决于硬件
  - 适合：中小分支、云端出口点
  
  部署方式：
    在 VMware vSphere 上
    在 KVM 虚拟化上
    在 Docker 容器中
    在公有云 VPC 中

混合部署：
  - 大分支用硬件 CPE
  - 小分支、临时办公点用软件 CPE
  - 数据中心出口用专业 CPE
  - 云端连接点用虚拟 CPE
```

## 部署与实施相关

### Q4: 迁移到 SD-WAN 需要多长时间？

**A：** 取决于企业规模和现有网络复杂度。

```
小企业（1-5 分支）：
  时间表：1-2 个月
  - 第 1 周：评估、采购
  - 第 2 周：概念验证（PoC）
  - 第 3-4 周：部署和测试
  - 第 5-8 周：迁移、优化
  
  成本：30-50 万元

中企业（5-30 分支）：
  时间表：2-4 个月
  - 第 1 个月：规划、PoC、采购
  - 第 2-3 个月：分批部署
  - 第 4 个月：优化和验收
  
  成本：200-500 万元
  
  策略：分批（先易后难）
    - 第一批：试点分支 2-3 个
    - 第二批：相似分支批量部署
    - 第三批：特殊分支定制

大企业（100+ 分支）：
  时间表：6-12 个月
  - 第 1 个月：规划和论证
  - 第 2-3 个月：PoC 和试点
  - 第 4-8 个月：分地区分批部署
  - 第 9-12 个月：优化和全量运维
  
  成本：2000-5000 万元
  
  难点：
    - 需要全球协调
    - 需要多个团队配合
    - 时区差异
    - 网络差异大

关键成功因素：
  ✓ 选择合适的迁移策略（大爆炸 vs 分批）
  ✓ 培训团队（运维人员需要学习新工具）
  ✓ 详细的回滚计划（万一出问题能快速回滚）
  ✓ 与厂商的紧密配合（不要自己瞎搞）
```

### Q5: 旧的网络设备（防火墙、路由器、WAN 优化器）怎么办？

**A：** SD-WAN 可以逐步替代这些设备。

```
迁移策略：

第一阶段（1-2 个月）：并行运行
  ┌─────────────────────┐
  │  新的 SD-WAN 网络   │ (建立新拓扑)
  │  ┌──────────────────┤
  │  └── SD-WAN CPE     │
  └─────────────────────┘
           ┌─────────────────────┐
           │  旧的 MPLS VPN      │ (继续运行)
           └─────────────────────┘
  
  这时期可以测试 SD-WAN，验证性能

第二阶段（2-3 个月）：流量逐步迁移
  一部分流量走 SD-WAN
  一部分流量仍走 MPLS
  
  关键应用和大流量先测试
  非关键应用后迁移

第三阶段（3-4 个月）：全量迁移
  所有流量都走 SD-WAN
  旧的 MPLS 可以保留作为备份
  或者完全下线

旧设备的去向：
  1. 保留作为备份（可选）
  2. 降配使用（小分支或临时用）
  3. 返回厂商换新（通常有折扣）
  4. 二手出售（回收成本）
  5. 废旧处理（联系专业回收机构）

注意：
  某些公司有合同限制
  确保与运营商确认退网计划
  避免被提前合约费用
```

## 性能与优化相关

### Q6: SD-WAN 会影响网络延迟吗？

**A：** 正常情况下不会，反而可能降低。

```
理论担忧：
  - SD-WAN CPE 增加了一层处理
  - 深度包检测（DPI）需要时间
  - 加密/解密有开销
  
  可能额外增加 1-3ms 的延迟

实际情况：
  SD-WAN 通常"降低"整体延迟，原因：
  
  1. 选择了更优的链路
     原本：上海 → 北京 → 深圳（200ms）
     现在：上海 ← 直连隧道 → 深圳（50ms）
     净收益：150ms 降低
     CPE 开销：1ms
     最终：降低 149ms
  
  2. 本地断网减少往返
     Office 365 访问不再经过总部
     北京总部出口延迟：100ms
     本地宽带出口延迟：30ms
     净收益：70ms
  
  3. 智能路由避免拥塞
     旧网络：被迫走拥塞链路，延迟 300ms
     SD-WAN：自动选择空闲链路，延迟 100ms
     净收益：200ms

性能监测：
  应该在上线前和上线后测量：
  - Ping 延迟
  - VoIP 一通通话延迟
  - Web 页面加载时间
  - 视频会议卡顿率
  
  如果有下降，立即调查原因：
  ✓ DPI 功能太重？→ 调整
  ✓ CPE 性能不足？→ 升级
  ✓ 隧道加密方式？→ 调整加密强度
```

### Q7: 可以在 SD-WAN 中运行 VoIP 吗？质量如何保证？

**A：** 完全可以，通常质量更好。

```
SD-WAN 对 VoIP 的优化：

1. 应用识别：
   VoIP 通话立即被识别
   → 自动提升到最高优先级（P1）
   → 获得带宽保证

2. 链路选择：
   VoIP 对延迟/丢包敏感
   → 只选择满足条件的链路
   → 可靠性大幅提升

3. QoS 保证：
   为 VoIP 预留带宽
   - 至少 100kbps/通话（单向）
   - 建议 200kbps/通话（双向）
   - 为高清 VoIP 可提升到 500kbps
   
   配置示例：
   ```
   voip {
     bandwidth-guaranteed = 1Mbps  // 10 通电话
     priority = P1
     jitter-tolerance = 20ms
     loss-tolerance = 1%
     allowed-paths = [preferred_wan, backup_wan]
     denied-paths = [4g_backup]  // 4G 太不稳定
   }
   ```

4. VoIP 特定优化：
   - RTP（实时传输）流量优化
   - 去除 RTP 头压缩（节省带宽）
   - 优先处理关键音频包
   - 自动编码率调整

实际体验：
  对比：MPLS VPN vs SD-WAN VoIP
  
  | 指标 | MPLS | SD-WAN |
  |-----|------|--------|
  | 延迟 | 50-200ms（固定） | 30-100ms（可优化） |
  | 丢包 | 0.1-1% | < 0.1% |
  | 音质 | 清晰 | 更清晰 |
  | 价格 | 贵 | 便宜 |
  | 可靠性 | 高 | 更高（多链路冗余） |

案例：某公司 VoIP 迁移
  背景：
    - 100 个分支
    - 原用企业 PBX（IP 电话系统）
    - 所有通话走 MPLS VPN（昂贵）
    - 客户投诉延迟和掉线
  
  问题分析：
    - MPLS VPN 有 2 条专线的瓶颈
    - 每个分支只有 2 条 10Mbps 线路
    - 不足以支持高峰时期的通话
  
  迁移方案：
    - 替换为 SD-WAN（多链路聚合）
    - 每个分支 2 条 50Mbps 宽带 + 4G 备份
    - 费用：降低 60%（从 MPLS 专线）
    - 质量：提升 40%（更多带宽，智能优化）
  
  结果：
    - 通话不再断线
    - 延迟更低
    - 成本更低
    - 客户满意度大幅提升
```

## 安全相关

### Q8: SD-WAN 相比传统网络的安全性如何？

**A：** 如果配置正确，安全性更高。但需要改变安全思路。

```
传统网络的安全模型：
  ┌──────────────────────────┐
  │   互联网                  │ (不信任)
  └──────────────────────────┘
           ↓ 防火墙 (厚重的城墙)
  ┌──────────────────────────┐
  │   公司内网                │ (信任)
  │   - 员工电脑              │
  │   - 服务器                │
  │   - 分支机构              │
  └──────────────────────────┘
  
  假设：内网很安全，只需要防止外部攻击

问题：
  ✗ 一旦进入内网，就有很多权限
  ✗ 横向移动容易（分支内网通常没有防火墙）
  ✗ VPN 连接的远程员工也信任（有风险）

SD-WAN 的安全模型：零信任（Zero Trust）
  ┌──────────────────────────┐
  │   互联网                  │ (不信任)
  └──────────────────────────┘
           ↓ 加密隧道 + 身份验证
  ┌──────────────────────────┐
  │   分支 A (不信任分支 B)  │
  │   - 可以访问总部          │
  │   - 不能直接访问分支 B    │
  └──────────────────────────┘
           ↓ 身份和应用层检查
           
  每一步都需要验证：
    - 你是谁？（身份）
    - 你用什么设备？（设备合规性）
    - 你要干什么？（应用和数据）
    - 你有权限吗？（授权策略）

SD-WAN 的安全优势：

1. 全加密
   所有分支间流量都通过 IPSec 隧道
   即使使用互联网链路，数据也是加密的
   
2. 细粒度访问控制
   不是"允许所有分支流量"
   而是"允许分支 A 的员工访问 ERP"
   
3. 应用层可见性
   知道谁访问什么应用
   而不仅仅是"IP 到 IP"
   
4. 云安全集成
   集成 Zscaler、Palo Alto 等安全服务
   所有互联网流量都过安全网关
   
5. 行为分析
   异常流量立即告警
   比如：员工凌晨 2 点上传大量数据？
   立即检测和阻止

配置示例：
  ```
  security {
    # 所有隧道使用 IPSec
    tunnel.encryption = AES-256-GCM
    tunnel.auth = SHA-384
    
    # 身份验证
    authentication = certificate + OTP
    
    # 访问控制
    rule {
      source = branch_shanghai_users
      destination = erp_system
      action = allow
      apply_dlp = true  // 数据丢失防护
    }
    
    rule {
      source = branch_shenzhen_users
      destination = erp_system
      action = deny  // 深圳分支不能访问
    }
    
    # 云安全
    internet-traffic {
      gateway = zscaler_cloud
      inspection = full
    }
    
    # 威胁防护
    threat-protection = enabled
    ips = enabled
    anti-malware = enabled
  }
  ```

安全的代价：
  ✗ 管理复杂性增加
  ✗ 需要更多的身份管理系统
  ✗ 某些内网访问可能被阻止（需要上报审批）
  ✓ 安全收益远大于成本
```

## 成本与 ROI 相关

### Q9: SD-WAN 真的能省成本吗？怎么计算 ROI？

**A：** 能省，但需要合理评估。

```
成本对比案例（100 个分支，3 年投资期）：

传统 MPLS VPN 方案：
  项目成本：
  ├─ 专线费用（100 分支 × 平均 15k/月 × 36 个月）：540 万
  ├─ MPLS 设备采购：50 万
  ├─ 设备维护（年 5%）：36 万
  ├─ 人工成本（2 个网络工程师 × 3 年）：120 万
  └─ 其他（光模块、电源等）：20 万
  
  总成本：766 万

SD-WAN 方案：
  项目成本：
  ├─ SD-WAN Controller（云服务）：50 万
  ├─ CPE 硬件采购（100 台 × 2k）：200 万
  ├─ 宽带费用（100 分支 × 平均 3k/月 × 36 个月）：108 万
  ├─ 4G 备份链路（50 分支 × 1k/月 × 36 个月）：18 万
  ├─ 人工成本（1 个网络工程师 × 3 年）：60 万
  ├─ 培训与咨询：30 万
  └─ 其他：20 万
  
  总成本：486 万

节省额：766 - 486 = 280 万（36% 节省）

其他收益（难以量化但实际存在）：
  ├─ 用户体验提升 → 生产效率提升 → 收益 2000+ 万
  ├─ 分支扩展时间从 2-4 周降到 2-3 天 → 加快上市
  ├─ 故障恢复时间从小时降到分钟 → 降低业务损失
  ├─ 云应用性能提升 → 应用迁移成本降低
  └─ 运维自动化 → 人力成本节省
  
  保守估计：额外收益 500+ 万

总 ROI = (280 万成本节省 + 500 万收益) / 486 万初期投资 = 159% (3 年)
年化 ROI: 约 53% / 年

现实中：
  中大型企业通常 1.5-2 年即可收回投资
  小企业可能需要 2-3 年
  前提是善于优化和充分利用 SD-WAN 的功能
```

### Q10: 中小企业能用 SD-WAN 吗？值不值得？

**A：** 可以，但要评估成本收益。

```
中小企业的特点：
  - 5-20 个分支
  - 总员工 100-500 人
  - 网络成本相对占比高

值不值得的决定因素：

1. 专线费用压力
   如果现在每年的专线费用 > 100 万
   → SD-WAN 节省约 40-50 万/年
   → 3 年回本，值得
   
   如果专线费用 < 50 万/年
   → 节省不多，可能不值得
   → 除非有其他急迫需求

2. 云应用使用程度
   如果已经大量用 Office 365、Salesforce 等
   → 本地断网的收益很大
   → 值得部署
   
   如果还在用本地 ERP
   → 云应用收益有限

3. 用户体验压力
   如果有大量投诉网络慢
   → SD-WAN 可以显著改善
   → 值得

4. 扩展计划
   如果计划在 3-5 年内扩展到 50+ 分支
   → 早期部署 SD-WAN 的成本摊销更低
   → 值得

中小企业的推荐方案：
  - 选择 SaaS 云控制器（无需维护）
  - 先试点 2-3 个分支（PoC）
  - 如果 ROI 正确，再全面部署
  - 可以考虑分期采购设备
  
  预算：50-100 万启动投资，3 年回本
```

## 厂商与选型相关

### Q11: SD-WAN 主要厂商有哪些？如何选择？

**A：** 主要有几个梯队。

```
第一梯队（Tier-1）：全功能，高端市场
  Cisco Meraki
    - 优点：功能完整、文档详细、支持好、合并了 Viptela
    - 缺点：价格较贵
    - 适合：大企业、需要高度集成
    - 代表用户：财富 500 强
  
  Fortinet FortiGate
    - 优点：集成防火墙、性能好、价格较合理
    - 缺点：配置复杂
    - 适合：需要安全的企业
    - 代表用户：中大企业
  
  VMware Velocloud（现已被 Broadcom 收购）
    - 优点：功能丰富、支持大规模部署
    - 缺点：价格贵
    - 适合：超大型企业、ISP

第二梯队（Tier-2）：性价比好，中端市场
  Versa Networks
    - 优点：可靠、性价比高、支持好
    - 缺点：品牌知名度低
    - 适合：中型企业、对成本敏感
  
  Cradlepoint
    - 优点：好用、支持移动办公
    - 缺点：功能不如一梯队全面
    - 适合：需要移动和遥远分支的企业

第三梯队（Tier-3）：专业或小众市场
  Arista CloudEOS
    - 优点：新技术、扩展性好
    - 缺点：相对新、用户较少
  
  Palo Alto Networks Prisma SD-WAN
    - 优点：安全能力强（自家防火墙集成）
    - 缺点：价格较贵
  
  国内厂商（云景、等）
    - 优点：本地化支持、价格便宜
    - 缺点：文档和国际支持不如国外厂商
    - 适合：仅在国内部署、对成本很敏感

选型建议：

  | 企业类型 | 推荐 |
  |--------|------|
  | 财富 500 强 | Cisco Meraki, Fortinet |
  | 大企业 (年收入> 50 亿) | Fortinet, VMware Velocloud, Versa |
  | 中企业 (年收入 10-50 亿) | Fortinet, Versa, 华为 |
  | 小企业 (年收入< 10 亿) | Cradlepoint, 国内厂商 |
  | 金融行业 | Palo Alto, Fortinet (安全要求高) |
  | 跨国企业 | Cisco, Broadcom, Versa (全球支持) |
  | 纯成本考量 | Versa, 国内厂商, 开源方案 |

决策流程：
  1️⃣ 确定预算范围
     < 200 万：小厂商或国内
     200-1000 万：Versa, Fortinet, 华为
     > 1000 万：Cisco, Broadcom, Palo Alto

  2️⃣ 列举必需功能
     写下你必须有的功能（不是"很好有"）
     - 全加密？
     - 云集成？
     - DPI？
     - 防火墙集成？
  
  3️⃣ PoC（概念验证）
     选 2-3 家，小规模试用
     评估易用性、性能、支持质量
  
  4️⃣ TCO 对比
     不仅看设备费用，还要看：
     - 订阅费（年费）
     - 支持费
     - 培训费
     - 预期使用年限
  
  5️⃣ 参考案例
     问厂商要与你相似的用户案例
     直接联系这些客户问体验

注意：
  ✓ 不要被厂商忽悠（都说自己最好）
  ✓ 关注支持质量而不仅仅是功能
  ✓ 考虑人才和生态（你能招到懂这个方案的人吗？）
  ✓ 避免过度采购（功能用不到，钱浪费了）
```

## 总结

这份 FAQ 覆盖了从基础概念到部署和选型的常见问题。如果你还有其他疑问，可以回到相关章节深入阅读。

**关键要点**：
- SD-WAN 不是替代 VPN，而是补充网络架构
- 部署不是一次性的，而是迭代优化的过程
- 安全需要主动设计，不是事后考虑
- ROI 通常在 1.5-2 年内实现
- 选择合适的厂商和方案比功能本身更重要
