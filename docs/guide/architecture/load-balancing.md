# 负载均衡深度解析：分散流量的艺术

## 导言

一个服务器能支持 10,000 并发。但你有 100,000 个用户。怎么办？

==负载均衡==就是把这 100,000 个用户的请求分散到 10 个服务器上。每个服务器只处理 10,000，完美！

这一章讲解负载均衡的原理、算法、故障转移和企业部署。

---

## 第一部分：负载均衡基础

### 为什么需要负载均衡？

```
没有负载均衡：
  所有 100,000 个用户 → 一个服务器
  └─ 服务器：我死了...
  
有负载均衡：
  100,000 个用户 
    ├─ 服务器 1（10K 个用户）
    ├─ 服务器 2（10K 个用户）
    ├─ 服务器 3（10K 个用户）
    ├─ ...
    └─ 服务器 10（10K 个用户）
  └─ 每个服务器都舒服
```

### 负载均衡的层次

```
第 4 层（传输层）：基于 IP + Port
  ├─ 性能最高（只看 IP 和端口）
  ├─ 能处理的并发最多
  ├─ 无法理解应用逻辑
  └─ 例：F5 BigIP、Nginx（四层）

第 7 层（应用层）：基于应用内容
  ├─ 性能一般（需要解析应用协议）
  ├─ 可以理解用户请求的内容
  ├─ 可以做更智能的路由
  └─ 例：Nginx（七层）、HAProxy

层级对比：
  4层：数据包 IP:192.168.1.1:8080 → 转发到服务器 1
  7层：HTTP 请求 GET /api/user → 根据路径转发到服务器 2
```

## 第二部分：负载均衡算法

### 算法 1：轮询（Round Robin）

```
轮流分配，每个服务器轮一个请求

请求序列：
  1 → 服务器 1
  2 → 服务器 2
  3 → 服务器 3
  4 → 服务器 1
  5 → 服务器 2
  ...

优点：
  [v] 简单
  [v] 公平（每个服务器负载相等）

缺点：
  [x] 不考虑服务器能力
  [x] 如果某个请求很重（耗 CPU），仍然均匀分配
```

### 算法 2：最少连接（Least Connections）

```
分配给当前连接数最少的服务器

服务器状态：
  服务器 1：50 个连接
  服务器 2：100 个连接
  服务器 3：30 个连接

新请求 → 分配到服务器 3

优点：
  [v] 更接近实际负载
  [v] 长连接不均匀时效果好

缺点：
  [x] 短连接时可能不均匀（快速创建和销毁）
```

### 算法 3：加权轮询（Weighted Round Robin）

```
根据服务器能力分配不同的权重

权重配置：
  服务器 1（高性能）：权重 50
  服务器 2（低性能）：权重 30
  服务器 3（低性能）：权重 20

分配：
  每 100 个请求中
  服务器 1 分配 50 个
  服务器 2 分配 30 个
  服务器 3 分配 20 个

优点：
  [v] 适应异构服务器
  [v] 可以手动调整权重

缺点：
  [x] 权重设置需要经验
```

### 算法 4：IP Hash

```
根据客户端 IP 计算 Hash，同一客户端总是转发到同一服务器

计算：
  Hash(客户端 IP) % 服务器数 = 服务器索引
  
  Hash(192.168.1.1) % 3 = 0 → 服务器 1
  Hash(192.168.1.2) % 3 = 2 → 服务器 3
  
同一 IP 的后续请求也去服务器 1

优点：
  [v] 会话保持（用户总是访问同一服务器）
  [v] 缓存友好（用户的缓存数据总在同一服务器）

缺点：
  [x] 如果某个 IP 的请求很多，会导致不均衡
  [x] 服务器数变化时，需要重新计算（大量会话转移）
```

## 第三部分：会话保持

### 问题：无状态 vs 有状态

```
无状态应用（REST API、Web 页面）：
  ├─ 每个请求独立，不依赖前面的请求
  ├─ 可以自由分配到任何服务器
  └─ 最容易负载均衡
  
有状态应用（购物车、游戏、登录状态）：
  ├─ 第 1 个请求：用户登录 → 服务器 1
  ├─ 第 2 个请求：添加商品 → 必须去服务器 1
  ├─ 第 3 个请求：查看购物车 → 必须去服务器 1
  └─ 如果分配到不同服务器，会丢失状态
```

### 解决方案 1：Session 复制

```
所有服务器的 Session 保持一致

方式：
  服务器 1：用户 A 登录 → Session 保存在自己
           → 广播给服务器 2、3
  
  服务器 2：收到用户 A 的请求 → 读取本地 Session
           → 找到用户 A 的登录状态
           → 处理请求 → 更新 Session → 广播更新

优点：
  [v] 对应用透明
  [v] 高可用（任何服务器宕机都没关系）

缺点：
  [x] 复制延迟（广播需要时间）
  [x] 增加网络流量
  [x] 扩展困难（服务器多了，复制成本高）
```

### 解决方案 2：集中式 Session 存储

```
所有服务器的 Session 存储在一个中央库

架构：
  服务器 1、2、3 
    → 都访问 Redis（或 Memcached）
    → 用户 Session 存在 Redis
    
优点：
  [v] 可扩展（添加服务器无压力）
  [v] 高可用（Session 独立存储）
  [v] 快速（Redis 性能高）

缺点：
  [x] 多了一个中央库（性能和成本）
  [x] 如果 Redis 故障，所有服务器都没有 Session
  └─ 解决：使用 Redis Cluster 或哨兵模式
```

## 第四部分：健康检查与故障转移

### 健康检查

```
问题：某个服务器故障了，负载均衡器还在给它分配请求

解决：定期检查服务器是否健康

方法 1：TCP 连接检查（最快）
  └─ 尝试连接 TCP 端口，成功 = 健康
  
方法 2：HTTP 检查
  └─ 发送 HTTP 请求（GET /health）
  └─ 如果返回 200，认为健康
  
方法 3：应用级检查
  └─ 检查数据库连接、缓存连接
  └─ 返回自定义的健康状态

配置：

  health_check {
    interval: 5s        # 每 5 秒检查一次
    timeout: 2s         # 超时 2 秒
    unhealthy_count: 3  # 3 次失败判定为不健康
    healthy_count: 2    # 2 次成功后恢复
  }
```

### 故障转移

```
服务器 1 故障

步骤：
  1. 健康检查发现服务器 1 无法连接
  2. 标记服务器 1 为"不健康"
  3. 停止分配新请求给服务器 1
  4. 已有连接怎么办？
     ├─ 选项 A：立即关闭（用户无法继续）
     ├─ 选项 B：等待连接自然结束（更友好）
     └─ 选项 C：迁移到其他服务器（复杂）
  5. 监控：向管理员发出告警
  6. 手动修复或自动重启服务器 1
  7. 一旦健康检查通过，重新分配流量

恢复速度：
  └─ 取决于健康检查的频率和阈值
  └─ 通常 10-30 秒内检测和转移
```

## 第五部分：企业部署

### 架构

```
互联网用户
  ↓
[VIP（虚拟 IP）：203.0.113.1]
  ↓
[负载均衡器（HAProxy/Nginx/F5）]
  ├─ 健康检查：定期 ping 后端
  ├─ 会话保持：IP Hash 或 Cookie
  └─ 日志：记录所有请求
  ↓ 分发
  ├─ [应用服务器 1：10.0.0.1]
  ├─ [应用服务器 2：10.0.0.2]
  ├─ [应用服务器 3：10.0.0.3]
  └─ [应用服务器 4：10.0.0.4]
  
每个应用服务器后面：
  └─ [数据库集群]（单独的 SPOF）
```

### 配置示例（Nginx）

```nginx
upstream backend {
  least_conn;  # 最少连接算法
  
  server 10.0.0.1:8080 weight=50;
  server 10.0.0.2:8080 weight=30;
  server 10.0.0.3:8080 weight=20;
  
  # 健康检查
  check interval=3000 rise=2 fall=3 timeout=1000 type=http;
  check_http_send "GET / HTTP/1.0\r\n\r\n";
  check_http_expect_alive http_2xx;
}

server {
  listen 80;
  server_name api.example.com;
  
  location / {
    proxy_pass http://backend;
    proxy_set_header X-Forwarded-For $remote_addr;
    proxy_set_header X-Real-IP $remote_addr;
    
    # 会话保持（基于 Cookie）
    sticky cookie srv_id expires=1h path=/ httponly secure;
  }
}
```

## 最佳实践

1. **冗余负载均衡器**
   - 至少两个负载均衡器（主备或主主）
   - 使用 VRRP 做虚拟 IP 转移

2. **监控负载均衡器本身**
   - 监控 CPU、内存、连接数
   - 告警：连接数接近上限

3. **会话管理**
   - 优先选择无状态设计
   - 如果必须有状态，用中央存储

4. **性能优化**
   - 启用 HTTP/2
   - 启用连接复用（Keep-Alive）
   - 启用压缩（gzip）

5. **安全**
   - 负载均衡器上部署 WAF（Web 应用防火墙）
   - 启用 SSL/TLS
   - 限制大请求（防止缓冲区溢出）
