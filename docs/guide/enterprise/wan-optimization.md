# WAN 优化完全指南：廉价高效的广域网

## 导言

WAN（广域网）的成本是LAN（局域网）的100倍，但性能却可能更差。这看起来不公平，但这就是现实。

==WAN 优化==技术通过压缩、缓存、流控等手段，让你用少一半的带宽做两倍的工作。

---

## 第一部分：WAN 的问题

### 带宽 vs 延迟 vs 丢包

```
WAN 三大杀手：

1. 带宽不足
   └─ 100 Mbps 的 WAN 听起来很快，但...
      分给 200 个员工，每人只有 0.5 Mbps
      
2. 延迟高（100ms ~ 500ms）
   └─ TCP 对延迟很敏感
      高延迟 → 吞吐量下降 → 用户体验差
      
3. 丢包率高（0.1% ~ 1%）
   └─ 看似很小的丢包，对应用的影响巨大
      1% 丢包 → TCP 吞吐量下降 30%

实际案例：
  总部和分支用 100 Mbps WAN 链接
  原始性能：100 Mbps
  实际吞吐（加上延迟、丢包影响）：20-30 Mbps
  原因：WAN 优化没做好
```

## 第二部分：WAN 优化技术

### 技术 1：数据压缩

```
原理：减少数据体积

文本数据压缩率：60-80%（GZIP）
├─ "aaaaaaaaaa" → "10a"（10 倍效率）
├─ HTML、JSON 都压缩效果好
└─ 但视频、图片（已压缩）效果差

配置：

Web 应用 → GZIP 压缩 → WAN → 分支 → 解压显示

好处：
  ├─ 100 Mbps 的链路，效果相当于 250 Mbps
  ├─ 成本最低（只需要 CPU，不需要硬件升级）
  └─ 兼容所有应用（对应用透明）

缺点：
  ├─ CPU 开销（压缩/解压需要 CPU）
  ├─ 有些数据（视频、图片）压缩效果差
  └─ 如果丢包，整个压缩数据块都要重新发送
```

### 技术 2：缓存

```
问题：员工 A 在分支下载 1GB 文件
      10 分钟后，员工 B 在同一分支也下载同一文件
      
没有缓存：
  员工 B 的文件又过一遍 WAN（又花 10 分钟）
  
有缓存（WAN 优化设备）：
  员工 B 的文件从本地缓存读取（1 秒）
  
效果：
  ├─ 10 倍的速度提升
  ├─ WAN 带宽节省 99%
  └─ 用户体验天差地别

缓存命中率：
  Office 环境：50-80%（很多人访问同样的文件）
  数据中心环境：30-50%（更多样化的访问）
```

### 技术 3：协议优化（TCP 加速）

```
问题：TCP 在高延迟网络中吞吐量下降

原理（传统 TCP）：
  发送数据 → 等待确认 → 再发送下一个
  └─ 延迟高时，大量时间浪费在等待上

TCP 加速原理：
  └─ 在 WAN 两端部署"TCP 代理"
     代理不等待远端确认，立即回复本地
     然后代理负责可靠传输到对方

配置：

总部的 WAN 优化器 ← TCP 加速 → 分支的 WAN 优化器
  ├─ 本地 PC 连到总部优化器（低延迟）
  ├─ 总部优化器连到分支优化器（高延迟）
  ├─ 分支优化器连到分支的应用服务器（低延迟）
  └─ 结果：TCP 吞吐量提升 5-10 倍

缺点：
  └─ 引入额外的设备（成本）
  └─ 如果中间链路出现故障，可能导致数据丢失
```

### 技术 4：去重（Deduplication）

```
原理：不发送重复的数据

场景：
  备份系统每晚备份 100GB 数据
  其中 70GB 是重复的（与前天相比）
  
传统备份：
  100 GB 过 WAN → 需要 10 小时（10 Mbps WAN）
  
有去重：
  只发送 30 GB（新增数据）→ 3 小时

工作流程：
  └─ WAN 优化器计算数据指纹（Hash）
     如果这个数据块之前见过，只发送指针
     对方根据指针恢复原数据

效果：
  ├─ 备份时间减少 70%
  ├─ WAN 带宽节省 70%
  └─ 对应用完全透明
```

## 第三部分：WAN 优化部署

### 架构

```
总部数据中心
  ├─ WAN 优化器（WOE - WAN Optimization Engine）
  ├─ 处理所有出站流量的压缩、缓存、去重
  └─ 接收分支的回传流量
  
分支办公室（x5）
  ├─ WAN 优化器
  ├─ 处理本地生成的流量
  ├─ 缓存从总部下载的数据
  └─ 加速 TCP

示意图：

[总部 WAN 优化器] ←-┬-→ [分支1 WAN 优化器]
                  ├-→ [分支2 WAN 优化器]
                  ├-→ [分支3 WAN 优化器]
                  ├-→ [分支4 WAN 优化器]
                  └-→ [分支5 WAN 优化器]

流量路径：
  分支PC → 分支WAN优化器 → WAN → 总部WAN优化器 → 总部应用
```

### 常见部署模式

```
模式 1：内联部署（Inline）
  所有流量必须过 WAN 优化器
  ├─ 优点：不需要修改网络配置
  ├─ 缺点：如果优化器故障，网络断掉
  └─ 需要配置故障转移
  
模式 2：透明代理模式（Transparent Proxy）
  WAN 优化器作为网关的一部分
  ├─ 优点：对终端用户透明
  ├─ 缺点：需要支持 WCCP 或其他重定向协议

模式 3：显式代理模式（Explicit Proxy）
  终端配置代理地址（类似 Web 代理）
  ├─ 优点：易于管理和监控
  ├─ 缺点：需要终端配置，不支持所有协议
```

## 第四部分：实战案例

```
场景：全球 20 个分支，总部在美国
      总 WAN 成本：每月 500 万人民币
      目标：减少 40% 的成本（每月 200 万）

现状：
  ├─ 每个分支 100 Mbps 专线
  ├─ 成本：100 Mbps 专线月费 25 万（按 20 个分支）
  ├─ 利用率只有 20%（大量浪费）
  └─ 用户投诉网络慢
  
问题：
  ├─ 专线虽然带宽足，但 TCP 效率低
  ├─ 没有缓存，同样的数据重复下载
  ├─ 没有去重，备份数据全部过 WAN
  └─ 没有压缩，文本数据没有压缩

解决方案：

Step 1：部署 WAN 优化器
  成本：$200K（一次性）+ 年维护费 $50K
  
Step 2：优化效果
  压缩：文本数据减少 70%
  缓存：热数据命中率 60%
  去重：备份数据减少 50%
  TCP加速：吞吐量提升 5 倍
  
  综合效果：带宽有效增加 3-5 倍

Step 3：降低成本
  原本需要 100 Mbps × 20 = 2000 Mbps
  优化后，相当于 400-600 Mbps（考虑中等效率）
  
  可以将链路降级到：
  ├─ 主要分支：50 Mbps（原来 100 Mbps）
  ├─ 次要分支：25 Mbps
  ├─ 小分支：10 Mbps
  
新成本：
  50×5 + 25×10 + 10×5 = 250 + 250 + 50 = 550 万月费
  节省：500 - 550 = -50 万？？？
  
  等等，这不对...应该是：
  原来 100×20 = 2000 Mbps
  优化后用 400 Mbps 实现相同性能
  
  成本节省：(2000-400)/2000 × 500万 = 400 万/月
  设备成本：200K 一次性 = 可在 1 个月 ROI
  
效果：
  ├─ 年度节省成本：4800 万
  ├─ 用户网络体验提升 5 倍
  ├─ 备份时间从 5 小时降到 1 小时
  └─ 总体 ROI：极好
```

## 最佳实践

1. **先优化后升级**
   - 软件优化（压缩、缓存）的效果 > 硬件升级
   - 成本更低，ROI 更快

2. **监控优化效果**
   - 部署后定期检查压缩率、缓存命中率
   - 确保设备正常工作

3. **分层部署**
   - 先在关键链路部署
   - 再逐步推广到其他链路

4. **测试故障转移**
   - 优化器故障时，流量是否能自动绕过
   - 定期演练
