# DDoS 防御完全指南：网络安全的终极挑战

## 导言

DDoS（分布式拒绝服务）攻击就像一个暴徒带 1 万个帮手砸你的店，让你无法正常营业。

了解 DDoS 的原理和防御方法，是现代网络管理员的必修课。

---

## 第一部分：DDoS 攻击类型

### 流量型攻击（容量耗尽）

```
目标：填满网络链路

方式 1：ICMP Flood
  └─ 攻击者发送大量 ICMP Echo Request（Ping）
     └─ 目标被 Ping 的回复淹没
     └─ 链路饱和

方式 2：UDP Flood
  └─ 发送大量 UDP 包到随机端口
     └─ 目标CPU 忙于处理

方式 3：SYN Flood
  └─ 发送大量 TCP SYN 包，不完成握手
     └─ 目标的 TCP 连接队列满了
     └─ 合法用户无法新建连接

流量规模：
  小型攻击：1 Gbps
  中型攻击：10-100 Gbps
  大型攻击：100-1000 Gbps（2020 年 AWS 遭遇 2.3 Tbps 攻击）
```

### 协议型攻击（资源消耗）

```
目标：消耗目标的计算资源（CPU、内存）

方式 1：DNS 反射攻击
  └─ 攻击者伪造源 IP 为目标，查询公网 DNS 服务器
  └─ DNS 回复发给目标
  └─ 放大效应：小请求导致大回复

方式 2：NTP 反射攻击
  └─ 类似 DNS，但用 NTP 时间协议
  └─ 放大倍数更大（60-100 倍）

方式 3：Smurf 攻击
  └─ 向广播地址发 ICMP，所有设备回复给目标
  └─ 已基本绝迹（现代网络不允许向广播地址发 ICMP）
```

### 应用层攻击（业务破坏）

```
目标：消耗应用资源（数据库查询、计算）

方式 1：HTTP Flood
  └─ 发送大量 HTTP 请求到 Web 服务器
  └─ 每个请求都是合法的，但数量庞大
  └─ 数据库连接池满，无法响应新请求

方式 2：Slowloris 攻击
  └─ 建立连接但故意很慢地发送 HTTP 请求
  └─ 耗尽服务器的连接资源

方式 3：SQL 注入（应用级）
  └─ 通过查询导致数据库 CPU 100%
```

## 第二部分：DDoS 防御方案

### 防御层次

```
第 1 层：链路层防护（ISP）
  └─ ISP 清洗中心：过滤明显的恶意流量
  └─ 通常免费或低成本

第 2 层：边界防火墙
  └─ 限制流量速率
  └─ 丢弃明显的攻击包（SYN Flood、ICMP Flood）

第 3 层：WAF（Web 应用防火墙）
  └─ 分析应用层流量
  └─ 检测和阻止应用层攻击

第 4 层：应用级防护
  └─ 应用自身的限流
  └─ API 速率限制
```

### 具体防御措施

#### 1. 限速和限流

```
在网络边界限制流量速率：

配置：
  rate-limit 100 Mbps for any source
  └─ 单个 IP 最多 100 Mbps
  └─ 超过部分被丢弃

效果：
  ✓ 阻止个别用户的异常流量
  ✗ 对分布式攻击无效（1 万台机器，每台 10 Mbps = 100 Gbps）
```

#### 2. 连接跟踪和状态清单

```
跟踪 TCP 连接状态，丢弃异常连接：

检查：
  ├─ SYN 包是否有对应的 ACK → 否 = 丢弃
  ├─ 数据包是否属于已建立的连接 → 否 = 丢弃
  └─ 连接是否超时 → 是 = 清除

效果：
  ✓ 防止 SYN Flood
  ✗ 对 UDP Flood 和应用层攻击无效
```

#### 3. GeoIP 过滤

```
对于国内网站，来自国外的流量可能是攻击：

配置：
  deny all traffic from non-China IPs  (如果只服务中国用户)

效果：
  ✓ 简单有效
  ✗ 误杀正常国外用户
  ✗ 攻击者可以从国外代理发起
```

#### 4. DNS 黑洞路由（Sinkhole）

```
问题：攻击导致合法用户也无法访问

黑洞路由方案：
  ├─ 在 DNS 返回一个假 IP（通常是 127.0.0.1）
  ├─ 用户无法访问网站
  ├─ 但服务器不再被淹没
  └─ 可以逐步恢复

缺点：
  ├─ 黑洞期间用户无法访问
  ├─ 不能彻底解决问题
  └─ 只是应急措施
```

#### 5. 流量清洗中心

```
第三方 DDoS 防护服务：

架构：
  用户 → Cloudflare/Akamai → 你的网站
         (清洗中心)
         
流程：
  1. 用户的流量先到清洗中心
  2. 清洗中心分析：合法流量 ✓ / 攻击流量 ✗
  3. 只转发合法流量到你的网站
  4. 攻击流量被丢弃

优点：
  ✓ 有效（拥有强大的分析能力）
  ✓ 黑洞期间用户仍能访问
  ✓ 专业团队 24/7 监控

缺点：
  ✗ 成本高（每月数万-数十万）
  ✗ 所有流量都要经过第三方（信任问题）
  ✗ 依赖第三方的 SLA
```

## 第三部分：应用级防护

### 频率限制（Rate Limiting）

```
检测异常的用户行为：

HTTP Flood 防护：
  ├─ 同一 IP 限制：100 请求/分钟
  ├─ 同一用户限制：1000 请求/分钟（基于登录状态）
  ├─ 全局限制：100K 请求/秒
  └─ 超过限制 → 返回 429 Too Many Requests

实现：
  nginx:
    limit_req_zone $binary_remote_addr zone=one:10m rate=10r/s;
    limit_req zone=one burst=20 nodelay;
```

### 验证码和 Challenge

```
Slowloris 防护：

当检测到异常模式（连接很多但数据很少）：
  1. 返回验证码 Challenge
  2. 真实浏览器能通过（JavaScript）
  3. 攻击脚本无法通过
  
原理：
  ├─ 验证码需要 JavaScript 执行
  ├─ 攻击脚本（如 curl）无法执行 JS
  └─ 给了真实用户和机器人的区分机会
```

### 黑名单和白名单

```
黑名单（已知的恶意 IP）：
  ├─ 来自已知的 DDoS 机器的 IP
  ├─ 来自专业防护组织（如 Shadowserver）的黑名单
  └─ 自己统计的异常 IP

白名单（可信的 IP）：
  ├─ 企业内网 IP
  ├─ 合作伙伴 IP
  ├─ CDN 节点 IP
  └─ 大型内容提供商 IP（Google、Baidu 等）

缺点：
  ├─ 需要持续更新
  ├─ 假阳性（好人被当坏人）
  └─ 攻击者可以轮换 IP
```

## 第四部分：实战案例

```
场景：你的网站遭受 10 Gbps DDoS 攻击

时间表：

T+0：发现攻击
  ├─ 监控告警：流量突增 10 倍
  ├─ 网站超时
  └─ 启动应急计划

T+5 分钟：初步响应
  ├─ 启用 GeoIP 过滤（只允许国内流量）
  ├─ 调用 ISP 的清洗服务
  ├─ 黑洞路由某些子网（暂时牺牲）
  └─ 网站部分恢复

T+30 分钟：中期应对
  ├─ 与 Cloudflare 签订应急合同
  ├─ 更换 DNS 指向 Cloudflare
  ├─ 等待 DNS 生效（TTL 可能需要 1 小时）
  └─ 攻击继续，但部分流量被清洗

T+2 小时：稳定
  ├─ DNS 已生效，大部分流量通过 Cloudflare
  ├─ 网站恢复正常
  ├─ 攻击继续，但已无效
  └─ 日志分析，追踪攻击源

后续：
  ├─ 与 FBI 合作追踪攻击者
  ├─ 改进防御措施
  ├─ 保留日志以备取证
  └─ 发布事件报告
```

## 最佳实践

1. **提前规划**
   - 制定 DDoS 应急预案
   - 与 ISP 和 CDN 提前沟通
   - 定期演习

2. **多层防护**
   - 不能依赖单一防护手段
   - 应用多个防御层次

3. **监控告警**
   - 实时监控流量异常
   - 尽早发现攻击

4. **日志保留**
   - 保留足够长的日志（用于事后分析和取证）
   - 帮助改进防御

5. **供应商选择**
   - 如果用第三方防护，选择信誉良好的（Cloudflare、Akamai、AWS Shield）
