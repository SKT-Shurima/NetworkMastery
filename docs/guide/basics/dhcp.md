# DHCP 深度解析：让网络配置自动化

## 导言

想象一个办公室有 500 台电脑。如果每台都要手动配置 IP 地址、网关、DNS，IT 部门会疯掉。

==DHCP（动态主机配置协议）==解决了这个问题：它像一个自动售票机，当你的设备连接到网络时，自动分配一个 IP 地址给你。

这一章讲解 DHCP 的工作原理、常见问题和企业级部署。

---

## 第一部分：DHCP 基础

### DHCP 是什么？

```
不用 DHCP：
  员工来公司
    → IT 手动给配置 IP：192.168.1.100
    → 配置网关：192.168.1.1
    → 配置 DNS：8.8.8.8
    → 配置子网掩码：255.255.255.0
    → 员工能上网了

用 DHCP：
  员工来公司
    → 接上网线
    → 完！自动配置好了！
```

### DHCP 四步握手（DORA）

```
D = Discover（发现）
O = Offer（提供）
R = Request（请求）
A = Ack（确认）

Step 1: DHCP Discover
  设备：我是谁啊？有人能给我分配 IP 吗？
  ├─ 以广播形式发送
  ├─ 源 IP：0.0.0.0（还没有 IP）
  ├─ 目标 IP：255.255.255.255（广播）
  └─ 包含自己的 MAC 地址

Step 2: DHCP Offer
  DHCP Server：我可以！我给你分配 192.168.1.100
  ├─ 源 IP：192.168.1.1（DHCP 服务器）
  ├─ 目标 IP：255.255.255.255（广播回复）
  ├─ 提供的 IP：192.168.1.100
  ├─ 子网掩码：255.255.255.0
  ├─ 网关：192.168.1.1
  ├─ DNS：8.8.8.8
  └─ 租期：24 小时

Step 3: DHCP Request
  设备：好的，我要这个 IP！（广播）
  └─ 确保所有 DHCP 服务器都知道我选了谁
    （如果有多个 DHCP 服务器，其他的就知道放弃）

Step 4: DHCP Ack
  DHCP Server：确认！你的 IP 租期从现在开始
  └─ 发送完整的网络配置

结果：设备现在有 IP、网关、DNS，可以上网了！
```

## 第二部分：DHCP 租期管理

### 租期是什么？

```
IP 地址不是"买来的"，而是"租来的"

租期过程：

获得 IP（t=0）
  ↓
使用 IP（t=0 ~ t=T）
  ↓
续约请求（t=T/2）← 租期到一半时主动续约
  ↓
租期更新（再延长 T）
  ↓
重复...

如果设备没有续约：
  t=T 时，IP 自动回收
  其他设备可以获得这个 IP
```

### 租期设置的艺术

```
太短（1 小时）：
  [v] IP 回收快，不浪费
  [x] 频繁续约，增加 DHCP 服务器负担
  [x] 网络波动时容易失去 IP

太长（30 天）：
  [v] 续约少，服务器压力小
  [x] 设备离职/移除后，IP 还被占着
  [x] 新员工来了，等不到新 IP

推荐做法：
  ├─ 办公网络：24 小时
  ├─ 访客网络：8 小时
  ├─ 数据中心（固定设备）：7 天或更长
  └─ 移动设备：4 小时
```

## 第三部分：DHCP 高可用部署

### 问题：DHCP 服务器宕机

```
场景：只有一个 DHCP 服务器

设备 A 来了，获得 IP 192.168.1.100
设备 B 来了，获得 IP 192.168.1.101
...
DHCP 服务器挂了！

新来的设备：求求给我分配 IP...
  └─ 没人回应，设备获不到 IP，无法上网

老设备续约：我的租期要到了...
  └─ 没人回应，租期过期后 IP 被回收

10 分钟后，整个网络都无法访问新设备或续约
```

### 解决方案：冗余 DHCP

```
方案 1：主从双机（Primary-Backup）
  
  主 DHCP：192.168.1.1
    ├─ 分配 192.168.1.100 ~ 192.168.1.200
    └─ 负责 80% 的流量
  
  从 DHCP：192.168.1.2
    ├─ 分配 192.168.1.201 ~ 192.168.1.250
    └─ 作为备份，负责 20% 的流量

优点：
  [v] 简单部署
  [v] 易于管理

缺点：
  [x] 范围要分配，容易冲突
  [x] 需要手动配置两个不同的池

方案 2：DHCP Failover（故障转移）

  两个 DHCP 服务器完全对称：
  ├─ 都知道所有可分配的 IP
  ├─ 使用 MCLT（Maximum Client Lead Time）同步
  ├─ 如果一个故障，另一个接管全部 IP
  └─ 设备无缝转移到另一个服务器

优点：
  [v] 完全冗余
  [v] 任何一个故障都无影响
  
缺点：
  [x] 配置复杂
  [x] 需要两个服务器性能对等
```

## 第四部分：DHCP 中继与跨域分配

### 问题：DHCP 是本地广播

```
DHCP Discover 是广播包（目标 IP：255.255.255.255）
广播包不能跨越路由器！

场景：
  分支 A（网段 10.0.1.0/24）：有 DHCP 服务器
  分支 B（网段 10.0.2.0/24）：没有 DHCP 服务器
  
  分支 B 的设备：我需要 IP...
  └─ 广播出去，但止步于路由器
  └─ DHCP 服务器收不到
  └─ 设备得不到 IP
```

### 解决方案：DHCP 中继代理

```
DHCP 中继代理（Relay Agent）：

分支 B 的设备 → 广播 DHCP Discover
                ↓
            路由器上的中继代理
                ├─ 听到广播
                ├─ 转换成单播
                ├─ 发到分支 A 的 DHCP 服务器
                ↓
            DHCP 服务器 → Offer
                ├─ 分配 IP：10.0.2.100
                ├─ 通过中继代理回复
                ↓
            分支 B 的设备获得 IP

配置：
  ip helper-address 10.0.1.10  (DHCP 服务器 IP)
```

## 第五部分：DHCP 常见问题与排查

### 问题 1：设备无法获取 IP

```bash
诊断步骤：

Step 1: 检查网络连接
  $ ethtool eth0
  → Link detected: yes

Step 2: 查看 DHCP 请求
  $ dhclient -v eth0 2>&1 | head -20
  → 应该看到 Discover、Offer、Request、Ack
  
  如果卡住了：
    DHCPv4 message type: DHCPDISCOVER
    DHCPv4 message type: DHCPOFFER
    → 没看到 Ack = 问题出在这里

Step 3: 检查 DHCP 服务器状态
  $ systemctl status isc-dhcp-server
  → 服务是否在运行？
  
  查看日志：
  $ tail -f /var/log/syslog | grep DHCP
  → 看是否有错误消息

Step 4: 检查网段配置
  $ cat /etc/dhcp/dhcpd.conf
  → 该网段是否被定义了？
  → 是否有足够的 IP 可用？

Step 5: 检查 DHCP 池是否用完
  $ show ip dhcp binding
  → 看已分配的 IP 数量
  → 如果等于池大小，说明已满

解决：
  ├─ 扩大 DHCP 池
  ├─ 减少租期（IP 更快回收）
  └─ 增加 DHCP 服务器
```

### 问题 2：DHCP Starvation 攻击

```
攻击原理：

攻击者不断伪造 MAC 地址，发送 DHCP 请求
  ├─ 伪造 MAC 1 → 获得 IP 1
  ├─ 伪造 MAC 2 → 获得 IP 2
  ├─ 伪造 MAC 3 → 获得 IP 3
  └─ ...直到 IP 池用完

结果：正常用户无法获取 IP

防护方法：
  ├─ DHCP Snooping：只信任特定交换机端口的 DHCP 包
  ├─ 在交换机上启用：ip dhcp snooping
  ├─ 指定信任端口：ip dhcp snooping trust
  └─ 限制学习速率：ip dhcp snooping limit rate 100

另一个防护：DAI（Dynamic ARP Inspection）
  └─ 检验 DHCP 和 ARP 的一致性
```

## 实战案例：企业 DHCP 部署

```
场景：500 人公司，5 个分支

需求：
  ├─ 每个分支有 100 台设备
  ├─ 分支间延迟 50ms（地理分散）
  ├─ 需要快速网络响应（不能等 DHCP 超时）

设计方案：

分支 1 - 5：都有本地 DHCP 中继代理
  ├─ 总部有主 DHCP 服务器
  ├─ 备用数据中心有从 DHCP 服务器
  └─ 故障转移：如果总部 DHCP 故障，自动用备用

DHCP 池配置：
  10.1.0.0/24 → 分支 1（分配 10.1.0.100 ~ 10.1.0.200）
  10.2.0.0/24 → 分支 2
  10.3.0.0/24 → 分支 3
  10.4.0.0/24 → 分支 4
  10.5.0.0/24 → 分支 5

租期：
  普通员工 PC：24 小时
  手机/平板：4 小时
  打印机等固定设备：7 天
  会议室临时设备：1 小时

成本：
  ├─ 一个 DHCP 服务器（虚拟化）：免费
  ├─ 各分支中继代理：已有（路由器自带）
  └─ 管理时间：每月 2 小时
```

## 最佳实践

1. **规划**
   - 为每个网段预估设备数量，IP 池要有 20% 冗余
   - 预留特殊用途的 IP 段（打印机、服务器等）

2. **部署**
   - 关键网络使用冗余 DHCP（Failover）
   - 所有分支都使用 DHCP 中继，指向中央服务器
   - 启用 DHCP Snooping 防止攻击

3. **监控**
   - 监控 DHCP 池使用率（预警：> 80%）
   - 监控 DHCP 服务器可用性
   - 定期审计 DHCP 分配日志

4. **故障转移**
   - 测试 DHCP 故障转移（每季度一次）
   - 确保两个 DHCP 服务器配置同步
   - 有应急预案（IP 池用完时的处理）
