# DNS 完全指南：互联网的电话簿

## 导言

你每天都在用 DNS，但你真的了解它吗？

==DNS（域名系统）==是互联网最重要的基础设施之一：它把人类可读的域名（www.google.com）转换成机器可理解的 IP 地址（142.250.185.46）。

这一章讲解 DNS 的工作原理、常见问题和企业级优化。

---

## 第一部分：DNS 基础

### 域名的层级结构

```
www.google.com
│   │      │
│   │      └─ 顶级域名（TLD）：.com
│   └─────── 二级域名：google
└─────────── 子域名：www

完整的域名叫做 FQDN（Fully Qualified Domain Name）

其他 TLD 的例子：
  ├─ .com：商业（美国或国际）
  ├─ .cn：中国
  ├─ .org：非营利组织
  ├─ .edu：教育机构
  ├─ .gov：政府
  └─ .io：技术公司最爱（岛屿国 IO 的域名，但被用作代表 Input/Output）

域名的右到左是从更大的范围到更小的范围
```

### DNS 查询过程（递归查询）

```
用户在浏览器中输入：www.google.com

Step 1: 本地递归解析器（通常是 ISP 的 DNS）
  浏览器：8.8.8.8，请帮我解析 www.google.com
  
Step 2: 递归解析器查询根域名服务器
  递归解析器：root server，www.google.com 的 IP 是多少？
  根服务器：我不知道，但我知道 .com 的 TLD 服务器在哪
  
Step 3: 递归解析器查询 TLD 服务器
  递归解析器：.com 服务器，www.google.com 的 IP 是多少？
  TLD 服务器：我不知道，但我知道 google.com 的权威服务器在哪
  
Step 4: 递归解析器查询权威服务器
  递归解析器：google.com 权威服务器，www.google.com 的 IP 是多少？
  权威服务器：是我管理的域名！www.google.com 的 IP 是 142.250.185.46
  
Step 5: 递归解析器返回给浏览器
  浏览器：142.250.185.46，好的！
  
浏览器：现在我用 142.250.185.46 去请求网页

总耗时：通常 10-100ms
```

## 第二部分：DNS 缓存与 TTL

### TTL（生存时间）

```
问题：如果每次都查询，网络开销太大

解决方案：缓存 + TTL

DNS 记录：
  www.google.com A 142.250.185.46 TTL=300
  
含义：
  ├─ 这个结果可以被缓存
  ├─ 但只能保存 300 秒（5 分钟）
  ├─ 5 分钟后过期，需要重新查询
  └─ 过期前都从缓存读取

缓存层次：
  浏览器缓存（最快）
    ↓
  操作系统缓存（macOS/Windows/Linux）
    ↓
  ISP 的递归解析器缓存
    ↓
  权威服务器（最慢，需要查询）
```

### 选择合适的 TTL

```
太短（60 秒）：
  ✓ DNS 变化立即生效
  ✗ 服务器负担大，缓存命中率低
  ✗ 用户解析延迟增加

太长（86400 秒 = 1 天）：
  ✓ 服务器压力小，性能好
  ✗ 如果 IP 更换，全球用户要等一天才能转到新 IP
  ✗ 故障恢复慢

推荐做法：
  ├─ 正常情况：3600 秒（1 小时）
  ├─ 容易变化的：300 秒（5 分钟）
  ├─ 稳定的基础设施：86400 秒（1 天）
  └─ 灾备时期：降低到 60 秒（快速转移）
```

## 第三部分：DNS 记录类型

### 常见记录类型

```
A 记录：IPv4 地址
  www.google.com A 142.250.185.46

AAAA 记录：IPv6 地址
  www.google.com AAAA 2607:f8b0:4004:80a::200e

CNAME 记录：别名（指向另一个域名）
  blog.example.com CNAME example.com
  └─ 访问 blog.example.com 时实际访问 example.com

MX 记录：邮件服务器
  example.com MX 10 mail.example.com
  ├─ 10 是优先级（数字越小优先级越高）
  └─ 有多个 MX 时，邮件系统按优先级尝试

TXT 记录：文本信息（用途多样）
  example.com TXT "v=spf1 ip4:192.168.1.1 ~all"
  └─ SPF：声明哪些 IP 可以代表我发送邮件（防垃圾邮件）

NS 记录：名称服务器
  example.com NS ns1.example.com
  └─ 声明谁是我这个域名的权威服务器

SOA 记录：起始权威记录（元数据）
  example.com SOA ns1.example.com admin.example.com ...
  └─ 包含：版本号、刷新间隔、重试间隔等信息
```

## 第四部分：DNS 高可用与容错

### 问题：DNS 单点故障

```
场景：只有一个 DNS 服务器

不幸的是...它宕机了

后果：
  ├─ 用户无法解析任何域名
  ├─ 即使网站服务器正常，也无法访问
  └─ 全球用户都受影响
```

### 解决方案：多 DNS 服务器

```
部署：至少两个 DNS 服务器

DNS 1：ns1.example.com（主机房）
DNS 2：ns2.example.com（备用机房，不同地理位置）

域名注册商配置：
  NS 记录 1：ns1.example.com
  NS 记录 2：ns2.example.com

客户端行为：
  查询 DNS 1
    ├─ 响应 ✓
    └─ 使用结果
    
  或者
  
  查询 DNS 1
    ├─ 无响应（超时）
    └─ 查询 DNS 2
       ├─ 响应 ✓
       └─ 使用结果

可靠性：
  ├─ 单个 DNS 宕机 = 99% 的查询成功（DNS 2 接管）
  ├─ 两个都宕机（极罕见）= 0% 的查询成功
  └─ 推荐：使用全球 DNS 服务（AWS Route53、Cloudflare 等）
```

## 第五部分：DNS 常见问题与优化

### 问题 1：DNS 污染与 GFW

```
症状：访问某些网站很慢或无法访问，但其他网站正常

原因：DNS 污染
  ├─ 某些 DNS 查询被中间人篡改
  ├─ 返回错误的 IP（或不返回）
  └─ 导致用户无法访问

表现：
  $ nslookup google.com 8.8.8.8
  └─ Google 的 DNS（8.8.8.8）会返回正确 IP
  
  $ nslookup google.com 114.114.114.114
  └─ 中国 DNS（114.114）可能被污染

防护方法：
  ├─ 使用国外的公共 DNS（8.8.8.8、1.1.1.1）
  ├─ 使用 DoH（DNS over HTTPS）加密查询
  ├─ 使用 VPN 改变出口 IP
  └─ 企业级：部署 DNS 代理，使用公共递归解析器
```

### 问题 2：DNS 放大攻击

```
攻击原理：

攻击者对 DNS 服务器发送查询：
  ├─ 伪造源 IP 为受害者 IP
  ├─ 查询大的记录（比如 ANY 记录，返回所有类型的记录）
  └─ 一个小请求（100 字节）引发大回复（1000 字节）

如果有 10,000 个被攻陷的 DNS 服务器同时做这个：
  ├─ 攻击流量 = 1000 × 10,000 = 1000 万字节/秒
  ├─ 受害者被洪泛
  └─ 网络瘫痪

防护方法：
  ├─ 限制 ANY 查询：禁用或限制返回大小
  ├─ 源地址验证：确保查询来自真实用户
  ├─ 速率限制：单个 IP 每秒最多 N 个查询
  └─ 部署 DDoS 防护设备
```

## 实战案例：企业 DNS 部署

```
场景：全球 5 个地区，需要快速 DNS 解析

需求：
  ├─ DNS 响应时间 < 50ms
  ├─ 99.99% 可用性
  ├─ 支持自定义域名

方案：

使用全球 DNS 服务（AWS Route53 或 Cloudflare）
  ├─ 自动在全球 150+ 个位置部署
  ├─ 智能路由：用户被自动定向到最近的 DNS 服务器
  ├─ 健康检查：如果后端服务故障，自动返回备用 IP
  └─ 月成本：$0.5（Route53）或 $0（Cloudflare 免费版）

优势：
  ├─ 无需自己维护 DNS 服务器
  ├─ 全球高可用
  ├─ 自动 DDoS 防护
  └─ 易于管理（Web 界面）

配置示例（Route53）：

  example.com A 记录：
    区域：美国东部
    值：10.0.0.1
    延迟路由策略：50ms（距离美国用户最近）
  
  example.com A 记录：
    区域：亚太地区
    值：10.0.1.1
    延迟路由策略：10ms（距离亚洲用户最近）
  
  用户在美国 → 自动解析到 10.0.0.1
  用户在亚洲 → 自动解析到 10.0.1.1
```

## 最佳实践

1. **设置合适的 TTL**
   - 不要一成不变，根据变化频率调整
   - 灾备期间降低 TTL

2. **多个 DNS 服务器**
   - 至少两个，不同地理位置
   - 推荐使用全球 DNS 服务商

3. **监控和告警**
   - 监控 DNS 响应时间
   - 告警：如果响应时间 > 200ms

4. **DNS 安全**
   - 启用 DNSSEC（防止篡改）
   - 限制区域转移（防止信息泄露）
   - 防止 DNS rebinding 攻击

5. **DNS 性能**
   - 启用 EDNS(0) 支持更大的回复
   - 缓存优化
   - 使用 DNS 预取（浏览器会预先解析常用域名）
