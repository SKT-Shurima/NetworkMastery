# 交换机与数据链路层：局域网的工作原理

## 导言

你有没有想过：为什么同一个办公室的电脑能互相通信，而不需要经过互联网？

答案就在==交换机==这个看似不起眼的黑匣子里。这一章讲解数据链路层（第2层）的核心工作原理——从 MAC 地址、交换机、VLAN 到网络环路防止。

---

## 第一部分：MAC 地址与链路层寻址

### MAC 地址是什么？

```
IP 地址用来找到全球的电脑（好比国家和城市）
MAC 地址用来找到局域网内的设备（好比楼号和房间号）

MAC 地址格式：48 位，通常写成 16 进制

例：08:00:27:00:4e:28
    ↑       ↑
    OUI     设备号
  (制造商ID)  (厂商给的唯一号)

有名的厂商前缀：
  ├─ Intel：00:1a:a0
  ├─ Apple：a4:5e:60
  ├─ Cisco：00:0c:29
  └─ 华为：98:39:be
```

### 以太网帧结构

```
|-------|-------|-------|-------|-------|-------|
|目标MAC| 源MAC | 类型  |  IP数据包  |   CRC检验  |
|6 字节 |6 字节 |2 字节 |  46-1500  |  4 字节    |
|-------|-------|-------|-------|-------|-------|

CRC：循环冗余检验，检测传输错误（不能纠正，只能检测）
```

### ARP（地址解析协议）

```
问题：我有 IP 地址，但我不知道对方的 MAC 地址

解决方案：广播 ARP 请求

Step 1: A 想给 B 发数据，知道 B 的 IP（192.168.1.100）
Step 2: A 广播 ARP 请求："谁有 192.168.1.100？请告诉我你的 MAC"
Step 3: 所有设备收到请求
Step 4: B 回复：我是 192.168.1.100，我的 MAC 是 08:00:27:00:4e:28
Step 5: A 缓存这个映射（ARP Cache），下次直接用 MAC 地址通信

ARP Cache 有 TTL（默认 4 小时），过期后需要重新 ARP
```

## 第二部分：交换机原理

### 交换机的工作流程

```
交换机就是个智能转接员：

数据包到达交换机端口 A
  ↓
检查帧的源 MAC、目标 MAC
  ↓
查看 MAC 地址表：目标 MAC 在哪个端口？
  ├─ 在表中 → 转发到那个端口
  ├─ 不在表中 → 广播到所有端口（除了来源端口）
  └─ 目标就是自己 → 丢弃（不转发）
```

### MAC 地址学习

```
初始状态：MAC 地址表是空的

第一个数据包到达：
  源 MAC：aa:bb:cc:dd:ee:01（端口 1）
  目标 MAC：aa:bb:cc:dd:ee:02
  
交换机学习：
  记录在表中：aa:bb:cc:dd:ee:01 → 端口 1
  
目标 MAC 不在表中，广播到除端口 1 外的所有端口

当 aa:bb:cc:dd:ee:02 回复时：
  源 MAC：aa:bb:cc:dd:ee:02（端口 2）
  交换机又学习：aa:bb:cc:dd:ee:02 → 端口 2
  
现在两个地址都在表中，后续通信可以直接转发（单播）
```

## 第三部分：VLAN（虚拟局域网）

### 为什么需要 VLAN？

```
场景：办公楼有 200 个员工，分成 4 个部门

不用 VLAN 的问题：
  ├─ 所有设备在同一个广播域
  ├─ 一个设备发送广播，所有 200 个都会收到
  ├─ 网络拥塞，安全问题（可以嗅探其他部门的流量）
  └─ 无法做访问控制（2 层没有 IP）

用 VLAN 的好处：
  ├─ 200 个设备分成 4 个 VLAN（每个 50 台）
  ├─ 广播只在自己的 VLAN 内传播
  ├─ VLAN 间通信需要通过三层（路由）
  ├─ 可以在路由器上设置防火墙规则
  └─ 不同部门的员工看不到彼此的共享文件夹
```

### VLAN 的实现

```
交换机端口标记：

端口 1-50：VLAN 10（市场部）
端口 51-100：VLAN 20（技术部）
端口 101-150：VLAN 30（财务部）
端口 151-200：VLAN 40（运营部）

Trunk 端口（连接到路由器或其他交换机）：
  ├─ 允许所有 VLAN 的流量通过
  ├─ 帧中会加上 VLAN ID（Tag）
  ├─ 格式：802.1Q 帧头 + VLAN ID
  └─ 路由器根据 VLAN ID 做转发
```

## 第四部分：生成树协议（STP）

### 为什么需要 STP？

```
问题：为了容错，有两条链接到网络

拓扑：
  交换机 A ─┬─ 链接 1 ─┬─ 交换机 B
            └─ 链接 2 ─┘

悲剧发生了：
  A 的广播包
    ├─ 从链接 1 → 到 B → B 转发回 A
    ├─ 从链接 2 → 到 B → B 又转发回 A
    └─ 结果：无限循环！广播风暴！

网络瘫痪！
```

### STP 工作原理

```
Step 1: 选举根桥
  ├─ 所有交换机比较 BID（Bridge ID）
  ├─ BID = 优先级 + MAC 地址
  ├─ 数值最小的成为根桥
  └─ 例：优先级都是 32768，则 MAC 最小的赢

Step 2: 计算路径成本
  ├─ 每条链接都有成本（速度越快，成本越低）
  ├─ 1 Gbps：成本 4
  ├─ 10 Gbps：成本 2
  └─ 100 Mbps：成本 19

Step 3: 选择端口角色
  ├─ Root Port（根端口）：到根桥的最佳路径
  ├─ Designated Port（指定端口）：链接上转发的端口
  └─ Blocked Port（阻塞端口）：不转发数据

结果：形成一棵树，没有环路！
```

### 实战案例：STP 收敛时间

```
场景：某条链接故障

问题：STP 需要 30-50 秒才能重新选择路径
      → 期间网络中断，应用超时

解决方案：
  ├─ RSTP（Rapid STP）：收敛时间 < 5 秒
  ├─ MSTP（Multiple STP）：支持多棵树
  └─ VST（VLAN Spanning Tree）：每个 VLAN 独立计算
```

## 第五部分：交换机故障排查

### 问题 1：某个端口无法通信

```bash
诊断：

1. 检查物理连接
   $ ethtool eth0
   → Link detected: yes（有物理链接）

2. 检查交换机端口状态
   需要登录交换机的管理界面（Telnet/SSH）
   show interface status
   → 应该显示 connected, up
   → 如果是 notconnect → 物理链接问题
   → 如果是 err-disabled → 端口被禁用

3. 检查 VLAN 配置
   show vlan brief
   → 该端口是否在正确的 VLAN？

4. 检查 MAC 地址表
   show mac-address-table dynamic
   → 连接的设备 MAC 是否在表中？

5. 测试连通性
   ping 同 VLAN 内的另一个设备
```

### 问题 2：广播风暴

```
症状：网络瘫痪，所有设备都收不到数据

原因：
  ├─ STP 未启用
  ├─ STP 参数配置错误
  └─ 环路存在

诊断：
  检查交换机 CPU 使用率
  show processes cpu
  → 如果 CPU 达到 100%，很可能是广播风暴
  
  查看数据包转发速率
  show interface | include Input queue|Output queue
  → 如果队列堆积，说明包处理不过来

解决：
  ├─ 立即关闭多余的链接（打破环）
  ├─ 启用 STP：spanning-tree mode rapid-pvst
  └─ 配置根桥：spanning-tree vlan 1 priority 4096
```

## 最佳实践

1. **VLAN 规划**
   - 按部门或功能划分，不要按地理位置
   - 管理 VLAN 和数据 VLAN 分离
   - 预留 VLAN 给未来扩展

2. **STP 配置**
   - 使用 RSTP 而非原始 STP（速度快 10 倍）
   - 手动指定根桥（不要让系统自动选）
   - 在关键链接启用 PortFast（终端设备）

3. **交换机冗余**
   - 关键交换机至少两台
   - 使用 VRRP/HSRP 做网关冗余
   - 定期测试故障转移
