# IPv6 完全指南：网络的未来已来

## 导言

IPv4 地址快用完了。地球上有 80 亿人，但 IPv4 只有 43 亿个地址。

==IPv6==用 128 位地址解决了这个问题。每个人都能分到海量的 IP。

这一章讲解 IPv6 的原理、与 IPv4 的共存，和部署难点。

---

## 第一部分：IPv6 基础

### IPv4 vs IPv6

```
IPv4：32 位地址
  └─ 总共 4,294,967,296 个地址
  └─ 全球用完了
  └─ 地址复用、NAT 成为常态

IPv6：128 位地址
  └─ 总共 340,282,366,920,938,463,463,374,607,431,768,211,456 个地址
  └─ 这是一个 39 位数！
  └─ 每个人可以分配海量地址

IPv4 地址：192.168.1.1
IPv6 地址：2001:0db8:85a3:0000:0000:8a2e:0370:7334

IPv6 简化写法：
  2001:db8:85a3::8a2e:370:7334  (:: 代表连续的 0)
```

---

## 第二部分：IPv6 地址分类

### 地址类型

```
单播（Unicast）：一对一
  ├─ 全局唯一地址（Global Unicast）：2000::/3
  │  └─ 可以在互联网上路由的地址
  ├─ 链路本地地址（Link-Local）：fe80::/10
  │  └─ 只在同一网段有效
  └─ 环回地址（Loopback）：::1
     └─ 类似 IPv4 的 127.0.0.1

多播（Multicast）：一对多
  └─ ff00::/8

任播（Anycast）：一对离你最近的一个
  └─ 用同一地址，多个服务器
  └─ 路由自动选择离你最近的一个
```

### IPv6 前缀长度

```
IPv4：192.168.1.0/24 表示 256 个 IP
IPv6：2001:db8:1::/48 表示 2^80 个 IP

典型分配：
  ├─ /48 给一个组织（一个大公司）
  ├─ /56 给一个分公司
  ├─ /64 给一个局域网
  └─ /128 给一个主机
```

## 第三部分：IPv6 通信

### SLAAC（无状态地址自动配置）

```
IPv4 需要 DHCP 来分配 IP

IPv6 可以自动配置：

步骤 1: 主机生成链路本地地址
  └─ fe80::（16 个 f）+ 主机 ID（通常来自 MAC 地址）
  └─ 例：fe80::1a2b:3c4d:5e6f:7a8b

步骤 2: 主机向网络询问前缀
  └─ 发送 Router Solicitation (RS)
  └─ 路由器回复 Router Advertisement (RA)
  └─ 包含：网段前缀（如 2001:db8:1::/64）

步骤 3: 主机自动生成全局地址
  └─ 使用收到的前缀 + 主机 ID
  └─ 例：2001:db8:1::1a2b:3c4d:5e6f:7a8b

结果：
  └─ 无需 DHCP，设备自动配置了可用的 IPv6 地址
```

## 第四部分：IPv4 和 IPv6 的共存

### 双栈（Dual Stack）

```
大多数现代网络同时支持 IPv4 和 IPv6

设备配置：
  IPv4：192.168.1.100
  IPv6：2001:db8:1::100

DNS 配置：
  www.example.com A 203.0.113.1 (IPv4)
  www.example.com AAAA 2001:db8::1 (IPv6)

访问优先级（RFC 6555）：
  ├─ 尝试 IPv6（更新的协议）
  ├─ 如果 IPv6 超时，并行尝试 IPv4
  └─ 返回最先成功的连接
```

### 隧道技术

```
企业可能只有 IPv4 网络，但想用 IPv6

6to4 隧道：
  IPv6 包被包裹在 IPv4 中
  
架构：
  IPv6 Host → [6to4 Gateway] → IPv4 Internet → [6to4 Gateway] → IPv6
             (IPv6 in IPv4)                  (IPv4 in IPv6)

缺点：
  ├─ 性能开销（额外的包装）
  ├─ 延迟增加
  └─ 管理复杂

替代：Hurricane Electric 等免费 IPv6 隧道服务
```

## 第五部分：IPv6 部署挑战

### 挑战 1：应用支持

```
许多老应用不支持 IPv6

例如：
  某家老银行的网银系统
  └─ 只能用 IPv4
  └─ 分配了 IPv6 但无法使用

企业用 IPv6 面临的困境：
  ├─ 新应用原生支持 IPv6
  ├─ 老应用只支持 IPv4
  ├─ 不能放弃老应用（业务关键）
  └─ 必须保持双栈
```

### 挑战 2：安全审计困难

```
IPv6 地址空间太大

传统防火墙规则：
  IPv4：白名单 10.0.0.0/8
  IPv6：白名单 2001:db8::/32（但潜在有 2^96 个地址）

问题：
  ├─ 地址扫描变成了不可能（扫完需要数百年）
  ├─ 攻击者可以选择任何地址，很难追踪
  └─ 无法通过 IP 过滤，必须用其他方式（SSL 证书等）
```

### 挑战 3：配置管理

```
从 /24 网段（256 个 IP）扩展到 /48 网段（2^80 个 IP）

问题：
  ├─ IPAM（IP 地址管理）工具需要升级
  ├─ 手动管理变成了不可能
  ├─ 必须自动化或使用新的工具
  └─ 培训成本高
```

## 第六部分：何时应该部署 IPv6？

### 适合现在部署的场景

```
✓ 新建网络：一开始就双栈
✓ 云环境：云厂商已经支持
✓ 移动网络：4G/5G 原生支持 IPv6
✓ 物联网：设备海量，必须用 IPv6
```

### 暂时不需要的场景

```
✗ 老旧企业网络：应用不支持
✗ 内网隔离（不连互联网）：暂无需求
✗ 资源有限：升级成本高
```

## 实战：检查你的网络是否支持 IPv6

```bash
# 检查是否有 IPv6 连接
ping6 ipv6.google.com

# 查看本地 IPv6 地址
ip addr show  (Linux)
ipconfig  (Windows)
ifconfig  (macOS)

# 测试 IPv6 网站
curl -6 ipv6.example.com  (强制用 IPv6)

# DNS 解析
nslookup example.com  # 看是否返回 AAAA 记录
```

## 最佳实践

1. **并不急着全量部署**
   - 双栈是长期方案
   - 逐步迁移应用

2. **从外向内**
   - 先让互联网访问支持 IPv6
   - 内网最后迁移

3. **地址规划**
   - 申请 /48（RFC 4291）
   - 预留空间给未来

4. **监控和测试**
   - 定期检查 IPv6 连通性
   - 测试双栈故障转移

5. **培训**
   - 运维团队需要学习 IPv6
   - 地址更长，诊断工具有区别
