# 网络事件应急响应：快速止血

## 导言

网络故障发生时，秒秒钟都是金钱。

一个 DNS 故障，全公司 1000 人无法工作——每分钟损失可能数万块。

==应急响应==就是一套有序的流程，让你能快速诊断和恢复。

---

## 第一部分：应急响应框架

### 四个阶段

```
Detection（检测）
  └─ 监控告警触发
  └─ 用户反馈
  └─ 目标：尽快发现问题

Response（应对）
  └─ 启动应急计划
  └─ 召集应急团队
  └─ 建立应急指挥中心（可能虚拟的）

Mitigation（缓解）
  └─ 快速止血（甚至不一定修复根本问题）
  └─ 恢复部分服务
  └─ 减少损失

Resolution（解决）
  └─ 修复根本原因
  └─ 完全恢复
  └─ 事后总结
```

---

## 第二部分：事前准备

### 应急计划清单

```
□ 应急联系表
  ├─ 网络管理员
  ├─ 系统管理员
  ├─ 数据库 DBA
  ├─ 业务负责人
  └─ 管理层
  
□ 故障恢复文档
  ├─ 常见故障的诊断步骤
  ├─ 紧急配置文件（备份）
  ├─ 快速回滚脚本
  └─ 厂商支持电话
  
□ 沟通计划
  ├─ 谁负责向管理层汇报
  ├─ 谁负责对外通告
  ├─ 沟通频率（每 15 分钟更新一次）
  └─ 状态页模板
  
□ 定期演习
  ├─ 每季度一次模拟故障演练
  ├─ 验证恢复流程是否可行
  └─ 发现潜在问题
```

## 第三部分：检测与初期应对

### 检测到故障

```
监控告警触发：
  "BGP 路由丢失"
  "DNS 查询失败"
  "网络延迟 > 500ms"

或用户反馈：
  "员工说网络很慢"
  "客户无法访问网站"

初期检查（< 5 分钟）：

Step 1: 确认问题的范围和严重程度
  ├─ 是全网问题还是某个部门？
  ├─ 有多少用户受影响？
  ├─ 业务影响有多大？
  └─ 能容忍多久的中断？

Step 2: 初期诊断
  ├─ ping 核心设备
  ├─ 检查 CPU/内存
  ├─ 查看最近的配置变更
  ├─ 检查最近的故障日志
  └─ 有没有用户不小心改了什么？

Step 3: 决定：升级还是等待？
  ├─ 问题严重 → 立即升级（启动应急计划）
  ├─ 问题轻微 → 继续诊断（可能自行恢复）
  └─ 不确定 → 有备无患，先启动应急计划
```

## 第四部分：快速缓解

### 常见问题的快速修复

#### 问题 1：链路故障

```
症状：某条 WAN 链路无法通信

快速缓解（不修复链路，但恢复服务）：

Step 1: 改变路由（如果有备用链路）
  └─ 手动改 BGP 权重
  └─ 流量自动转移到另一条链路
  └─ 用户恢复访问（可能更慢，但能用）

Step 2: 通知 ISP
  └─ 报修，预计多久恢复
  └─ 询问是否有其他链路

Step 3: 监控新路由的健康度
  └─ 确保转移后没有新问题

恢复后再做根本修复（修理链路、更换设备等）
```

#### 问题 2：DNS 故障

```
症状：DNS 无法解析域名

快速缓解：

Step 1: 切换 DNS 服务器
  └─ 如果主 DNS 故障，使用备用 DNS
  └─ 客户端改用 8.8.8.8 或 1.1.1.1（公网 DNS）
  └─ 大部分用户恢复
  └─ 企业内网可能仍无法解析（内部域名）

Step 2: 缓存 DNS 结果
  └─ 在关键服务器上本地配置 hosts 文件
  └─ example.com 127.0.0.1
  └─ 不依赖 DNS

恢复后再修复 DNS 服务器
```

#### 问题 3：服务器故障

```
症状：某台服务器宕机了

快速缓解（假设有冗余）：

Step 1: 从负载均衡器移除故障服务器
  └─ 新请求不再分配给它
  └─ 已有连接继续处理（如果允许）
  └─ 其他服务器容纳流量（可能性能下降）

Step 2: 告知用户可能变慢
  └─ 但能访问

恢复后再修复服务器（重启、硬件更换等）
```

## 第五部分：沟通和汇报

### 沟通矩阵

```
每 15 分钟向不同对象汇报：

向技术团队（每 5 分钟）：
  ├─ 最新诊断结果
  ├─ 采取的措施
  └─ 下一步计划

向业务负责人（每 15 分钟）：
  ├─ 当前状态（故障/部分服务/恢复中）
  ├─ 受影响用户数
  ├─ 预计恢复时间
  └─ 采取的措施

向管理层（每 30 分钟）：
  ├─ 简单一句话状态
  ├─ 业务影响（$）
  ├─ 预计恢复时间
  └─ 需要管理层做什么

向客户/外部（更新状态页）：
  ├─ "We are investigating a network issue"
  ├─ "Partial service restored"
  ├─ "Service fully restored"
  └─ 避免过多技术细节
```

## 第六部分：事后处理

### 事后总结（RCA - Root Cause Analysis）

```
故障原因：某员工在进行网络变更时不小心删了 ACL 规则

Root Cause：
  └─ 没有变更控制流程
  └─ 没有备份配置
  └─ 员工培训不足

改进措施：
  ├─ 实施变更控制（必须经过批准）
  ├─ 自动备份配置，支持快速回滚
  ├─ 员工培训：变更前要备份
  ├─ 配置管理系统（Git 版本控制）
  └─ 定期演习新流程

防止再次发生：
  ├─ 周期性审计
  ├─ 代码审查（配置也是代码）
  └─ 逐步推行变更（灰度发布）
```

---

## 最佳实践

1. **提前准备**
   - 不要等到故障时才想办法
   - 定期演习

2. **快速缓解，完全修复**
   - 优先止血（快速恢复部分服务）
   - 事后再做彻底修复

3. **清晰的沟通**
   - 不同对象不同语言
   - 诚实但不过度技术细节

4. **系统化改进**
   - 每次故障都是学习机会
   - 改进流程、工具、培训

5. **监控和告警**
   - 提前发现问题
   - 减少故障发现时间
