# 组播网络：一对多通信的高效方式

## 导言

想象你是一个电视台，需要向 1000 万观众同时播放同一个节目。

用==单播==（一对一）的方式需要 1000 万条独立的流——成本巨大。  
用==广播==（一对所有）不安全且浪费资源。

==组播==（多播，Multicast）解决了这个问题：一个发送者，多个接收者，只需要一条流！

---

## 第一部分：组播基础

### 单播 vs 广播 vs 组播

```
单播（Unicast）：一对一
  Server → PC A
  Server → PC B
  Server → PC C
  └─ 浪费带宽（重复发送）

广播（Broadcast）：一对所有
  Server → 所有设备
  └─ 不精确（即使不想要的设备也收到）
  └─ 无法跨越路由器

组播（Multicast）：一对多（选定的多个）
  Server → PC A, PC B, PC C（只有这些）
  └─ 高效（只发送一份）
  └─ 精确（只有订阅的设备接收）
  └─ 可以跨越路由器
```

### 组播地址

```
组播地址范围：224.0.0.0 ~ 239.255.255.255

分类：

224.0.0.0 ~ 224.0.0.255：链路本地组播地址
  └─ 不跨越路由器
  └─ 例：224.0.0.1（所有主机）

239.0.0.0 ~ 239.255.255.255：本地管理员组播地址
  └─ 企业内部用，不在互联网上用

例子：
  224.0.1.1：NTP（时间同步）
  224.0.23.12：IETF 会议音频
  232.0.0.0 ~ 232.255.255.255：源特定组播（SSM）
```

## 第二部分：组播的工作原理

### IGMP（网络组管理协议）

```
PC 想要接收组播流的过程：

Step 1: PC 向网络发送 IGMP Join 请求
  "我要加入 239.0.0.1 这个组"

Step 2: 交换机/路由器收到请求
  记录：PC A 想要 239.0.0.1

Step 3: 组播服务器发送流
  239.0.0.1 的流被转发到 PC A 所在的网络

Step 4: PC A 接收流
  └─ 所有发送到 239.0.0.1 的数据都到达 PC A

Step 5: PC 离开组
  "我不要 239.0.0.1 了"
  └─ 停止转发该流
```

### RPM（组播路由）

```
问题：如果发送者和接收者不在同一网络呢？

解决：组播路由器在网络间转发组播流

拓扑：

  [总部发送者]
        │ 组播流 239.0.0.1
        ↓
  [路由器 1]
        │ 转发流
        ├─ [分支 A 接收者]
        └─ [分支 B 接收者]

原理：
  ├─ 组播路由器维护组播转发表
  ├─ 当接收者 IGMP Join 时，路由器学习到有人想要这个组
  ├─ 路由器通知上游路由器也发送这个流
  └─ 形成一棵转发树

转发树示意：

  [发送者]
     │
   [核心路由器]
    ├── [分支 A 路由器] → [接收者 A1, A2]
    └── [分支 B 路由器] → [接收者 B1, B2]
```

## 第三部分：组播应用场景

### 场景 1：IPTV（网络电视）

```
场景：电信公司向 500 万用户同时播放足球比赛

用单播方式：
  需要 500 万条独立流 → 需要 500 万 Mbps 带宽
  成本：天文数字

用组播方式：
  所有用户加入同一个组 239.1.1.1
  只发送 1 条流 → 需要 5 Mbps 带宽
  成本：降低 100 万倍！
```

### 场景 2：视频会议

```
5 个分支参加视频会议，每个分支 20 人

用单播方式：
  分支 A 的 20 台设备都和 4 个其他分支通信
  流数：20 × 4 = 80 条
  
用组播方式：
  所有分支都加入 239.0.0.100
  只需 5 条流（每个分支一个发出去的）
  效率提升 16 倍
```

### 场景 3：服务器备份

```
数据中心有 1000 台服务器，需要同时做备份

用单播方式：
  1000 条独立的备份流 → WAN 链路饱和

用组播方式：
  所有服务器加入 239.0.0.200
  备份服务器只发 1 条流
  所有 1000 台服务器同时接收
  
效率：1000 倍
```

## 第四部分：组播的问题和限制

### 问题 1：不可靠性

```
组播使用 UDP，不保证送达

数据包丢失：
  ├─ 网络拥塞时可能丢包
  ├─ 接收者无法请求重传（不知道谁发的）
  └─ 某个接收者加入晚了，会丢失前面的数据

解决方案：
  ├─ 应用层实现 ACK 和重传
  ├─ 使用 FEC（前向纠错）恢复丢包
  └─ 某些应用（如 IPTV）可以接受少量丢包
```

### 问题 2：难以跨越公网

```
组播是为局域网设计的

互联网上的组播：
  ├─ 不是所有 ISP 都支持组播转发
  ├─ NAT 穿透困难（组播无法通过 NAT）
  ├─ 跨域组播路由困难（不同运营商之间）
  └─ 实际使用很少

通常解决方案：
  ├─ 在私有网络内用组播
  ├─ 跨越互联网用单播或 VPN
```

### 问题 3：管理复杂性

```
需要：
  ├─ 在路由器上启用 PIM（Protocol Independent Multicast）
  ├─ 配置 RP（Rendezvous Point，汇聚点）
  ├─ 或者用 SSM（Source Specific Multicast）
  └─ 监控组播转发树

对比单播路由，复杂度高 10 倍
```

## 第五部分：组播部署

### 企业组播网络

```
拓扑：

  [总部：组播源]
       │ PIM
  [核心路由器]
    ├─ [分支 A 路由器] - 启用 PIM
    ├─ [分支 B 路由器] - 启用 PIM
    └─ [分支 C 路由器] - 启用 PIM
    
配置要点：
  1. 在所有路由器启用 PIM-SM（Sparse Mode）
  2. 指定 RP（通常是核心路由器）
  3. 定义允许的组播地址范围
  4. 监控组播转发树

监控命令：
  show ip mroute      # 查看组播转发表
  show ip pim join    # 查看 Join 信息
  show ip igmp groups # 查看本地成员
```

## 最佳实践

1. **谨慎使用**
   - 只在需要的场景使用（视频、直播、备份）
   - 不要用来替代单播

2. **地址规划**
   - 使用 239.0.0.0/8（本地管理员范围）
   - 预留一些地址给未来

3. **监控**
   - 监控组播转发树的健康度
   - 告警：如果接收者无法加入组

4. **安全**
   - 限制可以发送组播的源
   - 防止恶意组播流

5. **容错**
   - 组播必须配合备用方案
   - 如果组播失败，切到单播
