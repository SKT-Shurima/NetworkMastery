# 流量整形与带宽管理：让网络有序运行

## 导言

想象一条高速公路在晚上 5 点下班时突然涌入 1 万辆汽车——结果就是堵车。

网络也有同样的问题。没有有效的流量管理，一个人在疯狂下载可能会拖垮整个公司网络。

==流量整形==和==带宽管理==就是网络上的"交通警察"，确保所有用户公平使用网络资源。

---

## 第一部分：为什么需要流量整形？

### 常见场景

```
场景 1：内存泄漏的应用疯狂上传日志
  ├─ 占用所有带宽
  ├─ 其他用户网络卡顿
  └─ 数据库备份迟到，恢复时间增加

场景 2：员工在上班时间看视频
  ├─ 视频使用 50% 的带宽
  ├─ 关键业务应用延迟高
  └─ 损失：生产力下降、用户体验差

场景 3：某个部门数据同步出问题
  ├─ 发送了 100 倍量的冗余数据
  ├─ WAN 链路饱和
  ├─ 所有分支网络变慢
  └─ 需要紧急处理
```

### 流量整形的好处

```
✓ 防止拥塞：限制单个用户或应用的最大速率
✓ 优先级控制：给关键应用保留带宽
✓ 公平分配：确保每个人都有合理的速度
✓ 可预测性：SLA（服务级别协议）更容易达成
✓ 成本控制：同样的带宽，能容纳更多用户
```

## 第二部分：流量整形算法

### 算法 1：令牌桶（Token Bucket）

```
工作原理：

想象一个桶，以固定速率往里放令牌

┌─────────────────┐
│   令牌桶        │
│   □ □ □ □      │  ← 令牌（代表可以发送多少数据）
│   □ □ □ □      │
│                 │
└─────────────────┘
   ↑ 每秒放入 100 个令牌
   
数据包要通过时：
  ├─ 看桶里有没有足够的令牌
  ├─ 有 → 拿走相应令牌，发送数据包
  ├─ 没有 → 等待（或丢弃）
  └─ 如果桶满了，多出来的令牌被扔掉

配置：
  速率：100 Mbps（每秒 100 个"令牌"，代表 100M 数据）
  桶大小：1000（最多能积累 1000 个令牌，允许短暂突增）

效果：
  平均速率：100 Mbps
  短暂突增：可以达到 1000 个令牌的数据（1 Gbps，但只能持续 10ms）
```

### 算法 2：漏桶（Leaky Bucket）

```
类似令牌桶，但反向：

┌─────────────────┐
│   流量桶        │  ← 数据进来
│   ║ ║ ║ ║      │
│   ║ ║ ║ ║      │
│   ╚═╩═╩═╩═     │  ← 固定速率漏出
└─────────────────┘

特点：
  ├─ 桶可以存储突发流量
  ├─ 但输出速率固定（恒定）
  └─ 不允许任何超速

与令牌桶的区别：
  令牌桶：允许短暂的突增（配合突发数据）
  漏桶：严格限制速率（更平滑）

应用：
  ├─ 令牌桶：适合互联网（需要应对突发流量）
  └─ 漏桶：适合 VoIP、视频流（需要恒定质量）
```

### 算法 3：加权公平队列（WFQ）

```
问题：令牌桶和漏桶都是"单一速率"

如果有多个用户呢？怎么分配他们的带宽权重？

加权公平队列解决方案：

为每个用户分配一个权重：

用户 A（关键应用）：权重 50
用户 B（普通用户）：权重 30
用户 C（低优先级）：权重 20

总权重 = 100

假设总带宽 100 Mbps：
  用户 A 获得：(50/100) × 100 = 50 Mbps
  用户 B 获得：(30/100) × 100 = 30 Mbps
  用户 C 获得：(20/100) × 100 = 20 Mbps

如果某个用户没用完自己的配额？
  └─ 剩余带宽分配给其他用户（按比例）

例：
  用户 A 只用了 30 Mbps（20 Mbps 未用）
  用户 B 想要 50 Mbps
  
  用户 B 可以获得：30 + (20×30/50) = 42 Mbps
```

## 第三部分：企业网络中的流量控制

### 场景 1：限制某个应用的速率

```
问题：监控系统（Prometheus）每分钟生成 100MB 日志
      导致 WAN 链路饱和

解决：限制 Prometheus 的出站速率

配置（以 Linux TC 为例）：

# 创建一个令牌桶规则，限制速率为 10 Mbps
tc qdisc add dev eth0 root tbf rate 10mbit burst 32kbit latency 400ms

# 结果：
# ✓ 平均速率 10 Mbps（可接受）
# ✓ 突发可以到 32 Kbit（很小，防止突增）
# ✓ 超过速率的包会被延迟或丢弃

效果：
  ├─ 之前：100 Mbps，链路饱和
  ├─ 之后：10 Mbps，其他应用可以正常工作
  └─ Prometheus 的上传时间变长（从 1 秒变 10 秒），但可接受
```

### 场景 2：给关键应用保证带宽

```
问题：VoIP 应用需要 2 Mbps 的稳定带宽
      但如果此时有 BitTorrent 下载，会被挤压

解决：使用优先级队列（Priority Queuing）

配置：

  队列 1（高优先级）：VoIP
    └─ 保证 2 Mbps，最多使用 5 Mbps

  队列 2（中优先级）：Web、邮件
    └─ 保证 20 Mbps

  队列 3（低优先级）：P2P、下载
    └─ 可用的剩余带宽（没有保证）

效果：
  ├─ VoIP 总是能获得至少 2 Mbps（清晰语音）
  ├─ 如果只有 VoIP，可以用到 5 Mbps（更好的质量）
  ├─ Web/邮件 获得 20 Mbps
  └─ P2P 使用剩余的带宽
```

### 场景 3：限制某个用户的速率

```
问题：某员工在疯狂下载（BT），占用所有带宽

解决：按用户划分带宽

配置（基于 IP 地址）：

  192.168.1.100（员工 A）：10 Mbps
  192.168.1.101（员工 B）：10 Mbps
  ...
  192.168.1.200（员工 Z）：10 Mbps

实现工具：
  ├─ Linux：iptables + tc (traffic control)
  ├─ 企业级：pfSense、Mikrotik RouterOS
  └─ 云平台：AWS CloudFront、Azure 限流

效果：
  ├─ 每个用户最多 10 Mbps
  ├─ 100 个用户，总带宽 = 100 × 10 = 1000 Mbps
  ├─ 但实际链路可能只有 200 Mbps（WAN 限制）
  └─ 系统会根据实际可用带宽分配
```

## 第四部分：常见工具

### Linux Traffic Control (tc)

```bash
# 查看当前的队列规则
tc qdisc show dev eth0

# 删除所有规则
tc qdisc del dev eth0 root

# 添加令牌桶，限制速率为 50 Mbps
tc qdisc add dev eth0 root tbf rate 50mbit burst 32kbit latency 400ms

# 查看实时的队列状态
tc -s qdisc show dev eth0

# 添加类别（更复杂的流量控制）
tc class add dev eth0 parent 1: classid 1:1 htb rate 100mbit
tc class add dev eth0 parent 1:1 classid 1:10 htb rate 50mbit  # VoIP
tc class add dev eth0 parent 1:1 classid 1:20 htb rate 30mbit  # Web
```

### 企业级工具

```
Cisco QoS:
  配置接口出方向的队列
  classify traffic by DSCP
  guarantee bandwidth for priority classes

Mikrotik RouterOS:
  队列管理系统（Queue Tree）
  基于 IP、Port、应用协议的流量分类
  支持 10+ 种队列算法

pfSense:
  流量整形（Traffic Shaping）
  VLAN 级别的带宽管理
  支持限流规则

AWS:
  CloudFront 限流
  API Gateway 限流
  NetworkFirewall 流量控制
```

## 实战案例：企业 WAN 优化

```
场景：
  总部和 5 个分支，共 1000 名员工
  WAN 链路总带宽：200 Mbps（很珍贵，成本高）
  需要：VoIP、视频会议、ERP、文件同步都能用

设计方案：

总带宽 200 Mbps 分配：
  ├─ VoIP：20 Mbps（保证）
  ├─ 视频会议：40 Mbps（保证）
  ├─ ERP 和关键业务：60 Mbps（保证）
  ├─ 文件同步：30 Mbps
  ├─ 备份：30 Mbps
  └─ 其他（Web、邮件）：20 Mbps

规则：
  Priority 1（最高）：VoIP - 保证 20 Mbps，最多 30 Mbps
  Priority 2：视频会议 - 保证 40 Mbps，最多 60 Mbps
  Priority 3：ERP - 保证 60 Mbps，最多 80 Mbps
  Priority 4：文件同步 - 保证 30 Mbps
  Priority 5（最低）：其他 - 使用剩余带宽

效果：
  ├─ VoIP 用户永远有清晰的语音（20 Mbps）
  ├─ 视频会议优先于普通流量
  ├─ 备份在非工作时间进行（避免与业务竞争）
  ├─ 同样的 200 Mbps，容纳 5 倍的用户
  └─ 用户体验一致，可预测
```

## 最佳实践

1. **测量和分析**
   - 了解你的流量特征（突发大小、平均速率）
   - 定期审计带宽使用情况

2. **分级分类**
   - 按应用或用户分类
   - 为不同类别设置不同的优先级

3. **监控和告警**
   - 监控队列深度（如果队列积压，说明配置不合理）
   - 告警：如果某个优先级的延迟超过阈值

4. **测试**
   - 在非工作时间测试流量整形规则
   - 确保不会意外阻止关键应用

5. **文档**
   - 记录所有的流量整形规则和原因
   - 便于后期调整和故障排查
