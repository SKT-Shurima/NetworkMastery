# TCP/IP 协议栈：现代网络的骨架

## 导言

如果 OSI 七层模型是网络通信的**理论框架**，那么 **TCP/IP 协议栈**就是**实际应用**。事实上，互联网上绝大多数的通信都遵循 TCP/IP 协议族。

这一章会深入讲解 TCP/IP 是如何工作的，以及为什么它能成为现代网络的标准。

---

## TCP/IP 协议族概览

### 什么是 TCP/IP？

**TCP/IP** 不是指某个单一协议，而是一个**协议族**的统称。它包含了数百个相关协议，最核心的两个是：
- **TCP（传输控制协议）** — 可靠的、连接导向的传输
- **IP（网际协议）** — 主要的网络层协议

### 分层模型

TCP/IP 采用**四层模型**（相比 OSI 的七层更简洁）：

```
┌──────────────────────────────────────────┐
│  应用层                                   │
│  HTTP, HTTPS, FTP, SMTP, SSH, DNS, DHCP │
├──────────────────────────────────────────┤
│  传输层                                   │
│  TCP, UDP, SCTP                          │
├──────────────────────────────────────────┤
│  网络层                                   │
│  IP (IPv4/IPv6), ICMP, IGMP, ARP        │
├──────────────────────────────────────────┤
│  链接层                                   │
│  以太网, WiFi, PPP                      │
└──────────────────────────────────────────┘
```

---

## 第一层：链接层（Link Layer）

链接层负责**物理网络的连接**。它直接处理硬件、网线、无线等媒介。

### 主要协议

**ARP（地址解析协议）**
- 功能：根据 IP 地址查询对应的 MAC 地址
- 场景：你知道对方的 IP，但需要找到他的 MAC 地址才能在局域网中通信
- 工作流程：
  ```
  1. 你要发给 192.168.1.100，但不知道他的 MAC
  2. 广播一个 ARP 请求："谁拥有 192.168.1.100？"
  3. 拥有该 IP 的设备回复："那是我，我的 MAC 是 aa:bb:cc:dd:ee:ff"
  4. 你记住这个对应关系，以后可以直接发送数据帧
  ```

**以太网（Ethernet）**
- 最常用的有线局域网标准
- 数据以**帧（Frame）**的形式传输
- 帧格式：`| 目标 MAC | 源 MAC | 类型 | 数据 | CRC |`

### 现实应用

在你的办公室里：
- 交换机（Switch）管理所有物理连接
- 通过 ARP，设备互相发现
- 所有的局域网通信都在这一层完成

## 第二层：网络层（Internet Layer）

网络层是 TCP/IP 协议族的**核心**。它实现了跨越不同网络的通信。

### IP 协议（IPv4）

**IP 报文的结构**：

```
┌─────────────────────────────────────────┐
│ 版本(4) | 首部长度(4) | 服务类型(8)    │
├─────────────────────────────────────────┤
│ 总长度(16)            | 标识(16)       │
├─────────────────────────────────────────┤
│ 标志(3) | 片偏移(13)                    │
├─────────────────────────────────────────┤
│ 生存时间(TTL)(8)      | 协议号(8)      │
├─────────────────────────────────────────┤
│ 首部校验和(16)                          │
├─────────────────────────────────────────┤
│ 源 IP 地址(32)                          │
├─────────────────────────────────────────┤
│ 目标 IP 地址(32)                        │
├─────────────────────────────────────────┤
│ 选项(可选)  | 填充                      │
└─────────────────────────────────────────┘
```

**关键字段说明**：

| 字段 | 长度 | 说明 |
|-----|------|------|
| 版本 | 4 位 | IPv4 (4) 还是 IPv6 (6) |
| 首部长度 | 4 位 | IP 报头的长度（以 32 位为单位） |
| 服务类型（ToS）| 8 位 | 指示优先级和延迟/吞吐量要求 |
| 总长度 | 16 位 | 整个 IP 报文的长度（包括报头和数据） |
| TTL（生存时间） | 8 位 | 每经过一个路由器减 1，到 0 时被丢弃 |
| 协议号 | 8 位 | 指示上层协议（6=TCP, 17=UDP, 1=ICMP） |
| 源/目标 IP | 各 32 位 | 发送者和接收者的 IP 地址 |

**IP 地址的含义**

IP 地址分为两部分：**网络部分**和**主机部分**

```
192.168.1.100 / 24
│   │   │  │     ↑
│   │   │  │     └─ /24 表示前 24 位是网络部分，后 8 位是主机部分
└───┴───┴──┘
  网络地址部分
  
网络地址：192.168.1.0
主机范围：192.168.1.1 ~ 192.168.1.254
广播地址：192.168.1.255
```

**子网掩码**（用于区分网络和主机）：
```
/24 = 255.255.255.0（24 个 1 和 8 个 0）
/16 = 255.255.0.0
/8  = 255.0.0.0
```

### ICMP 协议

**功能**：网络诊断和错误报告

**常见应用**：
- **Ping** — 检查主机是否可达
  ```
  $ ping google.com
  发送 ICMP Echo Request → 如果目标活着，回复 Echo Reply
  ```

- **Traceroute** — 追踪数据包的路由路径
  ```
  $ traceroute google.com
  显示数据包从你的电脑到 Google 服务器的每一跳
  ```

- **路由器间的错误通知**
  ```
  例如：目标网络不可达 → 发送 ICMP Destination Unreachable
  ```

### 路由（Routing）

路由是网络层最关键的功能。

**路由表的概念**：
```
$ route -n（在 Linux 中查看路由表）

目标网络        网关          接口
0.0.0.0         192.168.1.1   eth0      （默认路由：所有不匹配的包都发给网关）
192.168.1.0/24  直接连接      eth0      （本地网络：直接转发）
10.0.0.0/8      192.168.1.1   eth0      （其他网络：通过网关转发）
```

**路由工作流程**：
```
1. 路由器收到一个 IP 包，查看目标 IP
2. 在路由表中查找最匹配的条目（最长前缀匹配）
3. 根据条目确定：
   - 如果目标在直连网络，直接转发（ARP 查询 MAC）
   - 否则，转发给下一跳（网关）
4. 继续转发，直到到达目标
```

## 第三层：传输层（Transport Layer）

传输层为应用程序提供**进程间通信**的服务。

### TCP（传输控制协议）

**特点**：
- 面向连接（需要建立连接）
- 可靠传输（确保数据完整、有序）
- 流量控制（发送方适应接收方的速度）
- 拥塞控制（适应网络的承载能力）

**TCP 连接的三次握手**（建立连接）：

```
客户端                              服务器
  │                                   │
  │────── SYN (seq=x) ────────────→  │ (我想建立连接)
  │                                   │
  │ ←──── SYN-ACK (seq=y, ack=x+1) ── │ (好的，我也同意)
  │                                   │
  │────── ACK (seq=x+1, ack=y+1) ──→  │ (我收到了)
  │                                   │
  └────────── 连接建立 ────────────→  │
```

为什么是三次而不是两次？
- **第一次**：客户端告诉服务器"我要连接"
- **第二次**：服务器回复"我收到了"，但客户端还没确认服务器的应答
- **第三次**：客户端确认"我收到你的回复了"，双向通信才能开始

**TCP 四次挥手**（关闭连接）：

```
客户端                              服务器
  │                                   │
  │────── FIN (seq=x) ────────────→  │ (我要关闭连接)
  │                                   │
  │ ←──── ACK (ack=x+1) ────────────  │ (我收到了)
  │                                   │
  │ ←──── FIN (seq=y) ────────────────│ (我也关闭了)
  │                                   │
  │────── ACK (ack=y+1) ────────────→ │ (我收到了)
  │                                   │
  └────────── 连接关闭 ────────────→  │
```

### UDP（用户数据报协议）

**特点**：
- 无连接（直接发送数据，不需要握手）
- 不可靠（不保证到达和顺序）
- 低开销（报头只有 8 字节）
- 高速度

**适用场景**：
- 实时视频/音频通话（丢几帧没关系，但需要低延迟）
- 在线游戏（实时性更重要）
- DNS 查询（请求非常短小）
- DHCP（自动分配 IP）

**TCP vs UDP 的直观对比**：

```
TCP：像打电话
- 需要先拨号建立连接
- 对方必须接听，我们才能通话
- 如果噪音太大，可以要求重新说一遍
- 但建立连接需要时间

UDP：像大喇叭喊话
- 不需要建立连接，直接喊
- 对方听不听是他的事
- 信息可能丢失，但快速
- 适合不太重要的消息
```

## 第四层：应用层（Application Layer）

应用层是用户最直接感受到的层。

### HTTP/HTTPS（万维网）

**HTTP（超文本传输协议）**：
- 基于**请求-响应**模型
- 无状态（每个请求都是独立的）
- 使用 **TCP 端口 80**

**请求示例**：
```
GET /api/users HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Content-Type: application/json

[可选的请求体]
```

**响应示例**：
```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 42

{"users": [{"id": 1, "name": "Alice"}]}
```

**HTTP 状态码**：
- **2xx** (成功)：200 OK, 201 Created, 204 No Content
- **3xx** (重定向)：301 Moved Permanently, 302 Found, 304 Not Modified
- **4xx** (客户端错误)：400 Bad Request, 401 Unauthorized, 404 Not Found
- **5xx** (服务器错误)：500 Internal Server Error, 503 Service Unavailable

**HTTPS（HTTP + TLS）**：
- 在 HTTP 之下加一层**加密（TLS/SSL）**
- 保护数据的机密性和完整性
- 使用 **TCP 端口 443**

### DNS（域名系统）

**功能**：将域名翻译成 IP 地址

**工作流程**：
```
1. 你在浏览器输入：https://google.com
2. 浏览器向 DNS 递归服务器查询："google.com 的 IP 是什么？"
3. 递归服务器（通常是 ISP 提供的）不知道，向根域名服务器查询
4. 根服务器回复："去问 .com 的权威服务器"
5. 递归服务器向 .com 权威服务器查询
6. .com 权威服务器回复："去问 Google 的权威服务器"
7. 递归服务器向 Google 的权威服务器查询
8. Google 权威服务器回复："google.com 的 IP 是 142.251.41.14"
9. 递归服务器把结果返回给你的浏览器
10. 浏览器缓存这个结果，后续就可以直接用 IP 访问
```

**DNS 缓存的重要性**：
- **本地 DNS 缓存**（你的电脑/手机）— 几分钟到几小时
- **ISP DNS 缓存** — 可能几小时到几天
- **权威 DNS** — 根据 TTL 值设定

### SMTP/POP3/IMAP（电子邮件）

**SMTP（简单邮件传输协议）**：
- 功能：发送邮件
- 端口：25（未加密）、587（TLS）、465（SSL）

**POP3（邮局协议）**：
- 功能：接收邮件（将邮件下载到本地）
- 特点：下载后，邮件从服务器删除
- 端口：110（未加密）、995（SSL）

**IMAP（互联网消息访问协议）**：
- 功能：接收邮件（邮件保存在服务器上）
- 特点：可以在多个设备同步邮件
- 端口：143（未加密）、993（SSL）

### SSH（安全外壳）

**功能**：安全的远程登录和文件传输

**特点**：
- 基于密钥对的身份验证（比密码更安全）
- 所有通信都被加密
- 可以用于远程执行命令或传输文件（SFTP）

**端口**：22

**使用示例**：
```bash
ssh user@server.com          # 远程登录
scp file.txt user@server.com:/path/  # 复制文件到远程
sftp user@server.com        # 交互式 FTP 登录
```

## TCP/IP 协议族的优势

1. **开放标准** — 任何人都可以实现，促进了互操作性
2. **分层设计** — 每层相对独立，易于升级和扩展
3. **鲁棒性** — 几十年的生产验证，非常稳定
4. **可扩展性** — 从局域网到互联网，都能适应
5. **性能** — 经过长期优化，性能卓越

---

## 总结

TCP/IP 协议栈通过分层设计，让网络通信变得：
- **可管理** — 每层职责清晰
- **可互操作** — 统一的接口和协议
- **可扩展** — 新的应用可以基于现有基础构建

在后续章节中，我们会看到这个基础如何被企业网络、云网络和最终的 SD-WAN 所利用。
